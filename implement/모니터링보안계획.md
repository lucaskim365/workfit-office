# 📋 결재문서 모니터링 관리자/임원/일반 권한 분기 및 보안 해제 기능 설계서 (Option A)

본 설계서는 결재문서 모니터링 메뉴를 단일 경로(`/base/approval-monitor`)로 유지하되, 접속한 사용자의 권한 등급(**관리자 / 임원 / 일반**)에 따라 다이나믹하게 화면 구성과 조회 권한을 분기하고, 민감 정보 보호를 위한 2차 비밀번호 보안 잠금 해제 시스템을 구현하기 위한 상세 기술 명세서입니다.

---

## 1. 개요 (Introduction)
* **목적**: 전사 결재 문서 모니터링을 통한 실시간 업무 감사 및 추적성 확보.
* **보안 목표**: 
  * 일반 직원의 모니터링 화면 진입 원천 차단.
  * 권한 등급(관리자 vs 임원)에 따른 노출 데이터 범위 차별화.
  * 전사 문서 열람 시 민감 정보(인사, 급여 등) 유출을 방지하기 위한 **2차 비밀번호 검증 기반 본문 잠금해제** 구성.

---

## 2. 3단계 권한 아키텍처 및 메뉴 필터링 (3-Tier Authorization Architecture)

로그인 사용자의 권한을 **관리자 / 임원 / 일반** 3단계로 명확히 분리하여 접근 및 노출 범위를 제어합니다.
*(※ 구체적인 사용자 매핑 코드는 소스코드 내에서 직접 수정 및 기입 가능하도록 모듈화 설계)*

| 권한 등급 | 기술적 판별 기준 (예시) | 조회 가능 문서 상태 범위 | 상세 기능 권한 |
| :--- | :--- | :--- | :--- |
| **관리자 (System Admin)** | ID: `U012` (홍채원 사원) 등 | **전체 문서**<br>(진행중, 완료, 반려, 임시저장, 회수) | 전사 결재 흐름 감사,<br>2차 비밀번호 인증을 통한 본문 내용 조회 |
| **임원 (Executive)** | roleGroup: `OPERATOR` 등 | **승인 프로세스 유효 문서**<br>(진행중, 완료) | 진행 현황 모니터링,<br>2차 비밀번호 인증을 통한 본문 내용 조회 |
| **일반 임직원 (General)** | 위 두 조건 제외 전체 유저 | **접근 불가** (Menu 숨김) | 진입 시 권한 제한 경고 카드 노출 |

### 2.2 메뉴 노출 및 접근 제어 (GNB & Sidebar Guard)
* **GNB 메가 메뉴 및 사이드바**: `MENU_TREE` 내 `S_BASE_APMON` 화면 노출 조건에 유저 권한 체크 코드를 추가합니다. 일반 사용자는 메뉴 트리에서 해당 항목이 렌더링되지 않습니다.
* **라우터 레벨 보호**: 사용자가 브라우저 주소창에 `/base/approval-monitor` 주소를 수동 기입하여 접근하더라도, 스크린 컴포넌트 최상단에서 세션 권한을 체크하여 비인가자는 즉각 "접근 권한 제한" 템플릿(🔒 자물쇠 아이콘 및 경고 메시지)을 렌더링하도록 2중 가드를 적용합니다.

## 3. 동적 화면 구성 및 필터링 설계 (Dynamic View Routing)

사용자의 권한 등급에 따라 동일한 페이지 내에서 표시되는 탭 및 리스트 데이터 소스를 자동으로 필터링합니다.

```
[ApprovalMonitorScreen]
       |
       +---> 1. 관리자 뷰 (Admin) : 전체 탭 활성화 (전체, 진행중, 완료, 반려, 임시저장, 회수)
       |
       +---> 2. 임원진 뷰 (Exec)  : 제한 탭 활성화 (전체, 진행중, 완료) - 임시저장/회수/반려 필터링 생략
```

### 3.1 관리자용 모니터링 대시보드
* 전사 기안된 모든 상태의 결재 문서를 기안시간 역순으로 리스트업합니다.
* 전체, 진행중, 완료, 반려, 임시저장, 회수의 6개 상태 탭을 모두 제공하여 세부 필터링 감사를 지원합니다.

### 3.2 임원진용 모아보기 대시보드
* 실제 기안 결재선이 동작하고 있는 유효 문서만 조회 범위로 제한합니다.
* '임시저장', '회수' 및 '반려' 문서 탭은 숨김 처리되며, **진행중** 및 **완료** 상태의 문서들만 모아보기할 수 있습니다.

## 4. 2차 보안 잠금해제 기술 명세 (Security Unlock Spec)

민감한 문서를 보호하기 위해 기본값으로는 본문을 숨기고, 명시적인 비밀번호 입력 절차를 거친 경우에만 렌더링을 허용합니다.

### 4.1 UI/UX 흐름 (User Flow)
1. **기본(Locked) 상태**:
   * 목록에서 행을 클릭하면 우측 상세 영역에는 `결재선 진행 상태(타임라인)`만 렌더링됩니다.
   * 기안문 본문 카드 위치에는 회색 잠금 패널과 **"🔒 보안 잠금 상태입니다. 본문 내용을 확인하시려면 2차 비밀번호를 입력하세요."** 경고 및 **[본문 내용 잠금해제]** 버튼이 노출됩니다.
2. **비밀번호 입력 모달(Modal) 활성화**:
   * 버튼 클릭 시 화면 중앙에 미려한 패스워드 입력 모달이 팝업됩니다.
   * 암호를 입력하고 `Enter` 또는 `확인` 버튼을 누르면 내부 검증 로직이 실행됩니다.
3. **해제(Unlocked) 상태**:
   * 비밀번호 검증 성공 시 `isUnlocked` 상태값이 `true`로 토글되며 본문 카드(`ApprovalDocumentView`)가 정상 렌더링되어 품의 상세 내용 및 첨부파일을 확인할 수 있습니다.
4. **오토 락(Auto-Lock) 연동**:
   * **[보안 핵심]** 사용자가 목록에서 다른 결재문서 행을 클릭하여 선택 대상(`selId`)이 변경되는 즉시 `isUnlocked` 상태는 자동으로 `false`로 강제 초기화됩니다.
   * 즉, 하나의 문서를 해제했더라도 다른 문서를 볼 때는 매번 다시 비밀번호를 입력하게 설계하여 화면을 열어둔 채 자리를 비웠을 때 발생할 수 있는 보안 유출 사고를 방지합니다.

### 4.2 비밀번호 저장 방식
* 환경변수 파일(`.env.local` 또는 서버 환경 설정)에 다음 키값으로 2차 패스워드를 지정합니다.
  ```env
  VITE_MONITOR_UNLOCK_PW="admin1234"  # (기본값 설정, 필요시 자유롭게 수정)
  ```
* 프런트엔드 빌드 환경 및 Node.js 런타임에서 해당 환경변수를 참조하여 입력값과 일치 여부를 대조 검증합니다.
