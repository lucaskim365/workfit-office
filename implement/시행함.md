## 1. 기능 개요 및 정의
* **기능명:** 시행함 (Execution Box)
* **역할 분리 정의 (수신자 vs 참조자 vs 시행자):**
  * **수신자 (Recipient):** 문서가 종결(완료)된 후, 이를 공식적으로 통보받고 수신대장에 등록/관리하는 주체입니다. (업무 수행 추적 대상 제외, 조회용)
  * **참조자 (Cc):** 결재 진행 도중 또는 결재 완료 후, 진행 내용을 알고 있어야 하는 참고용 열람 대상자입니다. (업무 수행 추적 대상 제외, 조회용)
  * **시행자 (Executor / Execution Target):** 문서 결재가 최종 승인된 직후, 그 내용을 바탕으로 **실제 업무(집행, 실무 처리)를 의무적으로 처리해야 하는 실행 주체**입니다. (업무 수행 추적 대상, 상태 관리 가능)
* **시행 정의:** 
  * 최종 결재(승인)가 완료된 문서 중 실제 실무 집행/조치 작업이 필요한 대상 문서가 이관되어 진행 상태와 이력을 추적 및 완료 처리할 수 있는 보관함.
  * 열람 및 참조 용도인 '수신함', '참조함'과 달리 **상태 변화([대기중] -> [처리중] -> [시행완료])**와 실무 담당자의 **처리 Action**이 수반됨.

## 2. 주요 기능 및 프로세스 요구사항

### 2.1. 문서 자동 수신 및 이관 프로세스
1. **기안 시 시행자 지정:**
   * 기안자는 기안서 작성 단계에서 '시행자' (사용자 ID 또는 부서 ID)를 지정할 수 있습니다.
2. **결재 완료 시 자동 이관:**
   * 결재선의 모든 승인이 끝나 문서의 상태가 `완료`가 되는 시점에, 기안 시 지정된 '시행자'의 시행함 목록으로 해당 문서가 자동 수신되며 **[대기중]** 상태가 활성화됩니다.
   * 부서가 시행자로 지정된 경우, 해당 부서에 속한 모든 팀원의 시행함 목록에 해당 문서가 표시됩니다.

### 2.2. 시행 상태(Status) 수명 주기
문서는 시행함 안에서 다음과 같은 상태값을 가집니다.

* **[대기중] (Pending):** 결재가 완료되어 시행함에 처음 수신된 상태.
* **[처리중] (In Progress):** 담당자가 해당 업무 집행을 착수하여 [처리 시작] 또는 [내가 담당하기]를 지정한 상태.
* **[시행완료] (Completed):** 실제 조치(시스템 등록, 송금, 구매 등)를 마치고 완료 보고 처리를 마친 상태.

## 3. 상세 화면 및 UX 요구사항

### 3.1. 시행함 목록 화면
* **메뉴 구성:** 전자결재 메뉴 하위에 **[시행함]** 신규 메뉴를 생성합니다.
  * 메뉴 옆에 미처리(`대기중`, `처리중`) 상태인 문서의 개수를 카운트 배지로 표시합니다.
* **필터링 및 정렬:**
  * **상태 필터:** 전체, 대기중, 처리중, 시행완료 필터 탭 제공 (기본값은 미완료 건인 대기중/처리중 탭을 먼저 포커스).
  * **검색어:** 기안자, 기안부서, 문서번호, 제목 키워드 검색 지원.
  * **정렬:** 결재 완료일 역순(최신순) 정렬 제공.

### 3.2. 시행 문서 상세 및 처리 컴포넌트
* **시행 정보 영역:**
  * 문서 본문 상단 또는 결재선 옆 영역에 '시행 정보 영역'을 배치합니다.
  * 현재 시행 상태, 시행자 정보, 처리 일시, 처리 코멘트를 출력합니다.
* **사용자 액션 버튼:**
  * 로그인한 유저가 시행 대상자(또는 소속 부서원)인 경우에만 상태 변경 버튼을 노출합니다.
  * **[처리 시작] 버튼:** 상태를 `대기중` ➔ `처리중`으로 즉각 변경 처리합니다.
  * **[시행 완료] 버튼:** 클릭 시 시행 완료 팝업 모달을 표시합니다.
    * **시행 완료 팝업 필드:**
      * 시행완료일 (기본 오늘 날짜 제공, 변경 가능)
      * 처리 의견 (코멘트 기입란, 최대 500자)

## 4. 데이터 스키마 정의 (Draft)

기존 `ApprovalDoc` 내에 포함되거나 별도 매핑 테이블로 관리될 시행 이력 구조 정의입니다.

```typescript
export interface ApprovalExecution {
  /** 대상 결재문서 ID */
  docId: string;
  /** 기안 시 지정된 시행 대상 유형 ('USER' | 'DEPT') */
  targetType: 'USER' | 'DEPT';
  /** 시행 대상 ID (사용자 ID 또는 부서 ID) */
  targetId: string;
  /** 시행 처리 상태 */
  status: '대기중' | '처리중' | '시행완료';
  /** 실제 처리를 담당한 사용자 ID */
  executorId?: string | null;
  /** 처리 시작 일시 (ISO String) */
  startedAt?: string | null;
  /** 시행 완료 일자 (YYYY-MM-DD) */
  completedAt?: string | null;
  /** 시행 처리 의견 */
  comment?: string;
}
```

## 5. 예외 정책 및 상세 규칙
* **Q1. 시행자가 지정되지 않은 문서는?**
  * A. 최종 승인 시 시행함 이관 절차를 거치지 않고 단순 종결(완료)됩니다. (단순 수신 및 참조만 동작)
* **Q2. 반려되거나 회수된 문서는?**
  * A. 시행함 수신 대상에서 원천 배제됩니다.
* **Q3. 이미 완료된 시행 문서의 취소는?**
  * A. 완료 처리된 후에는 상태를 임의로 롤백할 수 없으며 필요시 코멘트 수정만 관리자 기능으로 제한 제공합니다.

## 6. 부서 시행자 지정 시 예외 처리 및 담당자 지정 정책

부서 단위(`DEPT`)로 시행자가 지정된 경우, 부서 전체 업무 누락을 막기 위해 부서원 전체의 시행함에 문서가 노출되나 중복 처리를 방지하기 위한 담당자 관리 체계를 둡니다.

### 6.1. 담당자 배정 기본 규칙 (신속성 확보)
* **[내가 담당하기] 기능:**
  * 부서 시행 문서가 [대기중] 또는 [처리중](담당자 지정 전) 상태일 때, 해당 부서원 누구나 상세화면에서 **[내가 담당하기]** 버튼을 클릭하여 스스로를 담당자(`executorId`)로 지정할 수 있습니다.
  * 본인이 담당자로 등록되는 순간 상태는 **[처리중]**으로 자동 전환되며, 타 부서원의 시행함 목록에서는 해당 건의 담당자 필드가 본인 이름으로 업데이트되어 노출됩니다.

### 6.2. 부서장 권한을 통한 강제 지정 및 변경 (통제 권한)
* **부서장 지정 권한:**
  * 로그인한 사용자의 직책이 해당 부서의 **부서장(부서 관리자)**일 경우, 담당자가 지정되지 않은 문서를 특정 부서원에게 강제로 지정하거나(배정), 이미 지정된 담당자를 다른 부서원으로 변경할 수 있는 **[담당자 지정/변경] Dropdown 메뉴**를 노출합니다.
  * 부서장이 담당자를 강제 배정할 시, 해당 부서원에게 업무 위임 로그가 기록되며 문서 상태는 **[처리중]**으로 전환됩니다.

### 6.3. 중복 처리 방지 조치
* 담당자가 지정된 문서에 대해서는 담당자가 아닌 일반 부서원의 경우 [시행 완료] 액션 버튼이 비활성화되며, 오직 **지정된 담당자**와 **부서장**만 시행 완료 처리를 할 수 있습니다.

## 7. 문서 조회 및 인쇄 시 메타 정보 개선 (시행일 / 보존연한 동적화)

기존 [ApprovalDocumentView.tsx](file:///c:/WorkFit/%EC%A0%84%EC%9E%90%EA%B2%B0%EC%9E%AC%EC%8B%9C%EC%8A%A4%ED%85%9C/workfit-office/src/modules/gw/approval/ApprovalDocumentView.tsx#L240) 상에서 하드코딩되어 제공되던 메타 정보를 기획 요구사항 및 시행 처리 데이터와 동기화되도록 개선합니다.

### 7.1. 시행일 동적 매핑
* **기존 정책:** 최종 결재 완료일(`doc.completedAt`)을 그대로 시행일로 출력.
* **개선 정책:**
  * 기안 시 시행자가 지정되지 **않은** 일반 문서는 종전대로 최종 결재 완료일(`doc.completedAt`)을 시행일로 판단하여 출력합니다.
  * 시행자가 지정되어 **시행 처리를 거친 문서**는 인쇄 및 조회 시 시행 완료 일시가 아닌 **시행 처리 시작 일시**(`ApprovalExecution.startedAt`)를 시행일 영역에 노출합니다. (상태가 '처리중'으로 변경되었을 때의 날짜)

### 7.2. 보존연한 동적 연동
* **기존 정책:** `'3년'`으로 화면 전체 문서 하드코딩 표기.
* **개선 정책:**
  * 서식 관리 도메인(`ApprovalForm`) 또는 문서 도메인(`ApprovalDoc`)에 보존연한(`preservationPeriod`) 항목을 설정할 수 있는 스키마 필드를 신규 추가합니다.
  * 문서 종류별(예: 기안 - 3년, 인사/재무 - 5년, 중요 계약서 - 영구 등) 양식에 정의된 보존연한 데이터 값을 동적으로 가져와 출력합니다. (설정값이 없을 시 기본값은 `'3년'`으로 대체 제공)

### 1단계: 도메인 스키마 추가 및 데이터 저장소 구축 (Data Layer)
* [ ] **보존연한 설정 추가:** `src/domain/approvalForm/schema.ts` (`ApprovalForm`)에 `preservationPeriod?: string` (기본값 `'3년'`) 필드 추가.
