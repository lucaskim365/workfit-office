> 작성일: 2026-07-15 · 대상: 메신저 첨부파일의 NAS 저장 가능성을 **end-to-end**로 검증하는 앱 내 개발용 테스트 페이지
> 선행 문서: `메신저_파일_NAS_관리_검토_보고서.md`(가능성 판단), `DEMO.md`(tus 파일서버 설계)

> 근거 코드: `src/app/App.tsx`(라우트), `src/app/menu-tree.ts`, `src/data/chatMessage/chatMessage.repo.ts`(현행 업로드), `.env.local`(env), `src/shared/lib/firebase.ts`

## 0. 결정 사항 (2026-07-15)

| 항목 | 결정 |
|---|---|
| 검증 범위 | **실제 NAS까지 end-to-end** (앱 → WebDAV 게이트웨이 → Synology WebDAV → 다운로드) |
| 테스트 페이지 위치 | **앱 내 개발용 라우트** (`/dev/file-lab`, 셸·로그인 재사용) |
| **NAS 백엔드** | **Synology WebDAV**(`https://workfit.synology.me:5006`, TLS). 로컬 tus 볼륨이 아니라 **WebDAV가 최종 저장소** |
| **인증 모델** | **2계층**: (1) 브라우저↔게이트웨이 = 앱 세션 JWT, (2) 게이트웨이↔Synology = 전용 서비스 계정 Basic/TLS. **브라우저는 DSM 자격증명을 절대 보지 않음** (§6) |

→ 목적: 메신저에 연결하기 **전에**, 앱→게이트웨이→NAS 업로드/다운로드 전체 경로와 **인증 경계**를 격리된 페이지에서 먼저 증명한다. 성공하면 동일 저장소 드라이버를 메신저 첨부(`uploadAttachment`)에 그대로 꽂는다.

> **RaiDrive 경로는 앱 인증이 아니다.** 스크린샷의 RaiDrive는 Windows 데스크톱에서 WebDAV를 Z: 드라이브로 마운트하는 **사람용 수동 관리** 수단이며, 브라우저 SPA(WorkfitOffice)는 마운트 드라이브에 접근할 수 없다. 앱의 파일 송수신은 아래 게이트웨이를 경유한다.

## 1. 배경 · 목표

메신저 첨부는 현재 Firebase Storage(`chat/{roomId}/…`)에 저장된다(`chatMessage.repo.ts:55-80`). NAS 전환의 핵심 미검증 구간은 **앱 브라우저 ↔ WebDAV 게이트웨이 ↔ Synology WebDAV**의 실제 왕복과 **인증 경계**다. 이 테스트 페이지로 다음을 증명한다.

1. 앱 화면에서 파일 선택 → **게이트웨이 업로드** → **Synology `messenger-files` 공유폴더에 실제 파일 생성**.
2. 업로드 응답으로 받은 **URL/ID로 미리보기(이미지)·다운로드**(원본 파일명 보존).
3. **대용량·이어올리기(tus resumable)** 동작(일시정지→재개, 탭 닫고 재개).
4. **저장소 드라이버 추상화**가 화면 코드 변경 없이 `firebase ↔ nas` 전환됨.
5. **2계층 인증**(Layer 1 앱 JWT · Layer 2 서비스 계정)이 성립하고, **브라우저에 DSM 자격증명이 노출되지 않음**.

### 목표가 아닌 것(이번 범위 밖)
- 메신저 UI 연결(첨부 전송 흐름 교체) — 본 검증 성공 후 별도 단계.
- 방 멤버십 기반 정식 인가, 운영용 HTTPS/역방향 프록시 하드닝, 쿼터·야간정리·백업 자동화.
- 기존 Firebase 첨부 데이터 마이그레이션.

## 2. 검증 아키텍처 (end-to-end)

```
┌─────────────────── WorkfitOffice 앱 (브라우저) ───────────────────┐
│  /dev/file-lab  (FileLabScreen)                                    │
│     │  파일 선택 · 업로드 · 진행률 · 목록 · 미리보기 · 다운로드     │
│     ▼                                                              │
│  fileStore.repo.ts  ← 저장소 드라이버 추상화 (firebase | nas)      │
│     │  uploadFile(file) → { url, name, size, mime }                │
└─────┼──────────────────────────────────────────────────────────────┘
      │  ▲ Layer 1 인증: 앱 세션 → 단기 JWT (Bearer). tus 업로드/다운로드
      ▼
┌────────────── WebDAV 게이트웨이 (Node, 서버측) ───────────────────┐
│  POST /api/upload (tus)  ·  GET /api/files  ·  GET /api/download/:id│
│  ① 앱 JWT 검증 + 앱 사용자·방 멤버십 인가                          │
│  ② 조립된 파일을 서버측에서 WebDAV PUT/GET (스트리밍)              │
│  ③ 메타 DB(sqlite: id·name·size·mime·앱uploader·roomId)            │
└─────┼──────────────────────────────────────────────────────────────┘
      │  ▲ Layer 2 인증: 전용 DSM 서비스 계정 Basic + TLS (자격증명은
      │    게이트웨이 env 에만 존재, 브라우저 미노출)
      ▼
┌──────────── Synology WebDAV  https://workfit.synology.me:5006 ─────┐
│  공유폴더 messenger-files/  (svc-fileapp 만 접근)                   │
└─────────────────────────────────────────────────────────────────────┘
```

- **앱**은 저장 위치·DSM 계정을 모른다. `fileStore.uploadFile()`가 유일하게 게이트웨이를 안다(정본 data-layer 패턴 준수).
- **게이트웨이**가 신뢰 경계다: 브라우저측은 앱 신원(JWT)만, NAS측은 서비스 계정만. 두 자격증명이 여기서 분리·교차된다.
- 업로드는 브라우저→게이트웨이 tus(이어올리기)로 받고, **게이트웨이가 서버측에서 WebDAV `PUT`**으로 Synology에 저장(대용량 스트리밍). 다운로드는 게이트웨이가 인가 후 WebDAV `GET`을 프록시.
- `DEMO.md`의 `X-Accel-Redirect`(nginx)는 로컬 볼륨 전송 최적화이며, WebDAV 백엔드에서는 게이트웨이 스트리밍 프록시로 대체(운영 최적화는 후속).

## 3. 구성요소 A — WebDAV 게이트웨이 (신규, 저장소 밖 `fileserver/`)

`DEMO.md` 기준 최소 구현이되, **저장 백엔드를 로컬 볼륨이 아니라 Synology WebDAV로** 둔다. 저장소 루트에 `fileserver/` 디렉터리 신설(앱 번들과 분리).

| 항목 | 내용 |
|---|---|
| 스택 | Node + Express + `@tus/server`(수신) + `webdav`(클라이언트) + sqlite(메타) |
| 업로드 | `POST/PATCH /api/upload` — tus(청크·이어올리기)로 수신 → 완료 시 서버측 WebDAV `PUT` |
| 목록 | `GET /api/files` — 메타 목록(id, name, size, mime, at, uploaderId, roomId) |
| 다운로드 | `GET /api/download/:id` — 인가 후 WebDAV `GET` 스트리밍 프록시 + `Content-Disposition`(원본명) |
| **저장 대상** | **Synology WebDAV** `WEBDAV_URL`의 `messenger-files/` 공유폴더 (로컬 임시 조립 후 업로드) |
| 인증 Layer 1 | `JWT_SECRET` 서명 토큰(Bearer) 검증 + 앱 사용자·방 멤버십 인가 |
| 인증 Layer 2 | `WEBDAV_USER`/`WEBDAV_PASS`(서비스 계정) — **게이트웨이 env 에만** |
| CORS | 앱 오리진(`localhost:3000`, 배포 도메인)만 허용 |
| 용량 | tus로 대용량 수신, WebDAV `PUT` 스트리밍(테스트 시 앱측 10MB 제한 완화 옵션) |

DoD: `curl`로 청크 업로드→`HEAD` 오프셋 확인→이어올림→**Synology 공유폴더에 파일 생성 확인**→다운로드 md5 일치.

## 5. 저장소 드라이버 추상화 (핵심 재사용 자산)

메신저와 공유할 **범용 파일 저장 추상화**를 신설한다. 검증 성공 시 이 파일 하나로 메신저까지 승격.

`src/data/fileStore/fileStore.repo.ts`
```ts
export type StoredFile = { url: string; name: string; size: number; mime: string };
export interface FileStore {
  upload(file: File, opts?: { scope?: string; onProgress?: (pct: number) => void }): Promise<StoredFile>;
}
// 드라이버 선택: import.meta.env.VITE_FILE_STORE_DRIVER (‘firebase’ | ‘nas’)
```
- `firebaseFileStore`: 현행 `uploadAttachment`의 Storage 업로드 로직 이식(재사용).
- `nasFileStore`: tus 클라이언트(`tus-js-client`)로 파일서버 업로드 → 다운로드 URL 조립.
- **화면(FileLabScreen)은 인터페이스만 소비** → 드라이버 교체가 화면에 무영향(보고서 §3의 seam 증명).
- 추후 메신저는 `chatMessage.repo.uploadAttachment`가 이 `fileStore.upload`를 호출하도록 위임만 하면 됨.

### env 추가
**앱(`.env.local`)** — 브라우저에 내려가는 값. **DSM 자격증명은 절대 넣지 않는다.**
```
VITE_FILE_STORE_DRIVER="nas"                 # firebase | nas
VITE_NAS_FILE_API="https://<gateway-host>/api"   # 게이트웨이(WebDAV 아님)
```
**게이트웨이(서버 env, 저장소 밖 `fileserver/.env`)** — 브라우저 미노출.
```
JWT_SECRET="***"                              # Layer 1 토큰 검증
WEBDAV_URL="https://workfit.synology.me:5006"  # Layer 2 대상
WEBDAV_USER="svc-fileapp"                      # 전용 서비스 계정
WEBDAV_PASS="***"
WEBDAV_BASE="messenger-files"                  # 접근 허용 공유폴더
CORS_ORIGIN="http://localhost:3000,https://<앱배포도메인>"
```

## 6. 인증 설계 — 2계층 (핵심)

**전제**: Synology WebDAV는 CORS 미지원 + Basic 인증이라 **브라우저가 직접 붙을 수 없다**(CORS 차단·DSM 자격증명 노출·앱 사용자↔DSM 신원 불일치). 따라서 게이트웨이를 신뢰 경계로 두고 인증을 두 층으로 분리한다.

### Layer 1 — 브라우저 ↔ 게이트웨이 (앱 신원)
- 앱은 자체 로그인(인메모리 세션, Firebase Auth 없음). 이 세션으로 **단기 JWT**를 발급해 게이트웨이에 Bearer로 전달.
- 게이트웨이가 JWT 검증 → **앱 사용자 식별 + (메신저 연결 시) 방 멤버십 인가**. 업로드·목록·다운로드 모두 이 검증 통과 필수.
- 검증 단계 최소안: 게이트웨이 `JWT_SECRET`로 서명한 앱 사용자 토큰(만료 포함). 다운로드도 공개 URL 대신 이 토큰 경로로만.

### Layer 2 — 게이트웨이 ↔ Synology (NAS 신원)
- **전용 DSM 서비스 계정 `svc-fileapp`** 하나로 WebDAV Basic 인증 over TLS(5006). administrators 제외, `messenger-files` 공유폴더 한 곳만 권한.
- 이 자격증명(`WEBDAV_USER`/`WEBDAV_PASS`)은 **게이트웨이 env 에만** 존재 → 브라우저·앱 번들에 절대 노출 안 됨.
- 게이트웨이가 파일 저장 시 `svc-fileapp` 권한으로 `PUT`, 다운로드 시 `GET`.

### 소유·감사 (결정: 단일 서비스 계정 채택)
- NAS 파일시스템 소유자는 전부 `svc-fileapp` → **"누가 올렸나"는 게이트웨이 메타 DB(uploaderId·roomId)로 추적**(앱 레벨 감사).
- 사용자별 DSM 계정 매핑은 채택하지 않음(브라우저 앱엔 이득 적고 12계정 프로비저닝·자격증명 보관 부담 큼). NAS 계정 단위 감사가 규제상 필수가 되면 재검토.

### Synology 측 사전 설정(체크리스트)
- [ ] WebDAV Server 패키지 설치, HTTPS 5006만 사용(TLS 1.2+, `workfit.synology.me` Let's Encrypt 인증서).
- [ ] `svc-fileapp` 계정 생성 → administrators 제외 → `messenger-files` 공유폴더만 읽기/쓰기.
- [ ] 공유기 5006 외부 노출은 게이트웨이 서버 IP로만(방화벽/허용목록), 원격관리 5006은 차단.
- [ ] (검증 후) Hyper Backup으로 `messenger-files` 외부 백업.

> **데모 한계 명시**: Layer 1 JWT는 자체 로그인 기반이라 진짜 Firebase Auth/`request.auth`가 아니다. 방 멤버십 인가는 메신저 연결 단계에서 완성(보고서 §4-#3). 이번 검증은 **경계 구조가 성립함**을 증명하는 데 목적이 있다.

## 7. 단계별 로드맵

| Phase | 산출물 | 완료 기준(DoD) |
|---|---|---|
| **0. Synology 준비** | WebDAV 패키지·`svc-fileapp`·`messenger-files` 공유폴더·인증서 | RaiDrive/curl로 `svc-fileapp` WebDAV PUT/GET 성공, 권한 격리 확인 |
| **1. 게이트웨이 스캐폴딩** | `fileserver/`(tus 수신 + `webdav` 클라이언트 + sqlite + JWT) | curl로 청크 업로드→이어올림→**Synology 공유폴더에 파일 생성**→다운로드 md5 일치 |
| **2. 저장소 추상화** | `fileStore.repo.ts` + `firebase`/`nas` 드라이버, env 스위치 | tsc 통과, 드라이버 전환이 화면 무변경으로 동작 |
| **3. 테스트 페이지** | `FileLabScreen` + `/dev/file-lab` 라우트, 업로드/진행률/목록/미리보기/다운로드 | 앱→게이트웨이→**Synology 저장 확인**→앱에서 다운로드 원본 일치 |
| **4. 이어올리기·대용량** | tus 일시정지/재개, 탭 닫고 재개, WebDAV PUT 스트리밍 | 100MB+ 업로드 중 재개 성공, 처음부터 안 올라감 |
| **5. 인증 경계 검증** | Layer 1 JWT(만료·무효 거부) + Layer 2 자격증명 미노출 확인 | 무효 토큰 거부, 브라우저 네트워크·번들에 DSM 계정 미노출 확인 |
| **6. 검증 리포트** | 결과 정리 + 메신저 연결 판단 | 아래 §8 체크리스트 전항 통과/이슈 기록 |

- Phase 0(Synology)·1(게이트웨이)은 인프라, 2·3(앱 코드)은 병렬 착수 가능(3은 2 이후).
- 검증: 각 Phase 후 `/run`(run-app) 또는 로컬 구동으로 `/dev/file-lab` 직접 진입해 육안 확인 + Synology File Station으로 실파일 확인.

## 8. 검증 시나리오 (체크리스트)

- [ ] 이미지 업로드 → 목록 썸네일 미리보기 정상.
- [ ] 일반 파일(zip/pdf) 업로드 → 다운로드 시 **원본 파일명·내용 일치**(md5).
- [ ] 한글 파일명 보존(`Content-Disposition` UTF-8).
- [ ] 대용량(≥100MB) 업로드 진행률 표시 + **일시정지→재개**.
- [ ] 탭 닫고 같은 파일 재업로드 시 **이어올리기**(tus localStorage).
- [ ] 드라이버 env를 `firebase`↔`nas`로 바꿔도 **화면 코드 변경 없이** 동작.
- [ ] **Synology `messenger-files` 공유폴더에 실제 파일 생성 확인**(File Station).
- [ ] **Layer 1**: 토큰 없이/만료·위조 토큰으로 업로드·다운로드 시 거부(401/403).
- [ ] **Layer 2**: 브라우저 네트워크 탭·번들에 DSM 계정(`svc-fileapp`)·비번이 **일절 없음**.
- [ ] `svc-fileapp`가 `messenger-files` 외 공유폴더엔 접근 불가(권한 격리).
- [ ] 게이트웨이/Synology 미가용 시 앱 에러 메시지 노출(무한로딩 방지).

## 9. 리스크 · 대응

| 리스크 | 영향 | 대응 |
|---|---|---|
| 브라우저→WebDAV 직접(CORS·자격증명 노출) | 근본적 불가 | **게이트웨이 경유 강제**(§6). 브라우저는 게이트웨이만 호출 |
| 자체 로그인 ↔ Layer 1 JWT 불일치 | 인증 실패 | 데모 발급 토큰으로 성립 확인, 방 멤버십 정식 인가는 후속 |
| DSM 서비스 계정 유출 | NAS 침해 | env·시크릿에만 보관, `messenger-files` 단일 폴더로 권한 최소화 |
| Synology 5006 외부 노출 | 공격면 | 게이트웨이 IP 화이트리스트·방화벽, TLS만, 원격관리 차단 |
| WebDAV PUT/스트리밍 대용량 | 게이트웨이 부하·지연 | 스트리밍 전송, 검증엔 허용. 운영 최적화(캐시/프록시)는 후속 |
| NAS 단일 장애점 | 데이터 손실 | RAID + Hyper Backup(`messenger-files`) |
