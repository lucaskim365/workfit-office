## 0. 개요

- 본 문서는 179화면이 실제로 다루는 데이터를 분석해 도출한 **Firestore 컬렉션 카탈로그 + 관계 + 채번 + 상태머신**이다.
- 설계는 [[DB_이관_대비_설계원칙.md]]의 6원칙을 준수한다: **정규화 / ID 참조(FK) / 안정 PK / 얕은 구조 / zod 단일 스키마 / RDB 친화 타입.**
- 컬렉션명은 영문 camelCase 복수형으로 통일(RDB 테이블명 직결).
- 총 컬렉션: **마스터 24 · 트랜잭션 41 · 집계 9 · 시스템 9 = 약 83개** (헤더-라인 분리·서브컬렉션 포함).

### 분류 체계

| 분류 | 정의 | 쓰기 빈도 | 이관 시 |
|---|---|---|---|
| **마스터(Master)** | 변경 드문 기준정보, 참조 대상 | 낮음 | 단순 테이블 |
| **트랜잭션(Transaction)** | 업무 이벤트, 채번·상태머신 보유 | 높음 | 테이블 + FK |
| **집계(Aggregate)** | 읽기전용 파생(리포트·대시보드) | 배치/트리거 | 뷰/머티뷰 |
| **시스템(System)** | 권한·로그·설정 등 횡단 | 가변 | 테이블/설정 |

## 1. 정규화·통합 결정 (중복 엔티티 단일화)

여러 모듈에 흩어진 동일 개념을 **단일 컬렉션**으로 통합한다. 이게 모듈 간 참조 일관성의 핵심.

| 통합 컬렉션 | 흩어져 있던 화면/모듈 | 통합 방식 |
|---|---|---|
| `items` | base·prod·mat·sales·qual 전부 | **단일 품목 마스터.** type(원자재/부자재/반제품/완제품)로 구분. 최다 참조 |
| `vendors` | base(거래처)·sales(고객)·equip(보전업체)·mat(협력사)·qual | **단일 거래처 마스터.** type(매입/매출/외주/보전/부품/교정) + 매출거래처는 여신·등급 필드 |
| `equipments` | base·prod·equip | **equip 모듈이 정본**, 나머지는 FK 참조. `equipmentUnits`·`equipmentSpecs` 서브컬렉션 |
| `lots` | prod(LOT계보)·mat(stockLots)·qual(추적) | **단일 LOT 마스터** + 재고관점 필드. 계보는 `lotGenealogy`, 이동은 `lotMovements` |
| `gauges` | equip(계측기)·qual(measuringGages) | **단일 계측기 마스터** + 검교정·MSA 연계 |
| `defectCodes` | base(불량항목)·qual(불량코드) | **단일 불량코드 마스터.** 발생 건은 `defectRecords`(prod) |
| `routings`(+`routingSteps`) | base·prod | **헤더+스텝 분리** 단일화 |
| `users` | base(사용자)·sys(관리자) | **단일 사용자.** role/level 필드로 일반↔관리자 구분 |
| `downtimes` | prod(비가동)·equip(downtimeLog) | **단일 비가동 원장.** category로 BM/PM/준비/자재대기 구분 |
| `inspectionItems`(품질) ↔ `equipCheckItems`(설비) | qual·equip | **이름 충돌 → 분리.** 품질검사항목과 설비점검항목은 별개 |

## 2. 컬렉션 카탈로그

> 표기: `필드 — 타입 — 설명`. PK=문서ID, FK→대상, ⬡=enum, 🔢=채번, ⚑=상태머신.

### 2.1 공통 마스터 (Cross-cutting)

#### `companies` (회사정보) — 시스템
`name, enName, bizNo, corpNo, ceo, founded(date), bizType, bizItem, corpKind⬡(법인|개인), fiscalYearStart(int), tel, fax, email, homepage, addr, sysName, reportHeader, docFooter, maskSensitive(bool), active(bool)`

#### `sites` (사업장)
`PK, name, kind⬡(본점|제조장|물류장), addr, tel, mgr→users, active(bool)`

#### `users` (사용자)
`PK, empNo, name, dept, position, roleGroup→roleGroups, email, level⬡(슈퍼관리자|시스템관리자|운영관리자|일반), status⬡(사용|잠금|미사용), twoFa(bool), ipRule, lastLogin(datetime)`

#### `roleGroups` (권한그룹)
`PK(code), name, desc, members→users[], menuPermissions: { [menuId]: {access:bool, edit:bool} }`

#### `vendors` (거래처/협력사) ★통합
`PK(code), name, type⬡(매입|매출|외주|보전|부품|교정), bizNo, ceo, manager, tel, email, addr, use(bool)` <br/>
매출 전용: `creditLimit(int), grade⬡(A|B|C)` · 보전 전용: `sla, contractStart/End(date), deliveryScore, qualityScore`

#### `items` (품목) ★최다 참조
`PK(code), name, spec, unit⬡(EA|L|Box|kg|SHT|ton…), type⬡(원자재|부자재|반제품|완제품), itemGroup⬡(원자재|화학약품·가스|포토레지스트|포장·부자재), safetyStock(int), price(int), use(bool), remark`

#### `commonCodes` (공통코드) — 모든 enum 원천
`PK(groupCode+code), groupName, name, order(int), use(bool), regBy`

#### `menus` (메뉴)
`PK(id), name, icon, order(int), use(bool), url, parentId→menus` (3단계: 모듈→그룹→화면)

## 5. 공통 규약 (모든 컬렉션 적용)

- **공통 필드:** `id`(PK), `createdAt`/`updatedAt`(ISO string), `createdBy`/`updatedBy`(→users), `siteId`(→sites, 멀티사업장 분리), `_v`(스키마 버전).
- **타입 규약:** 날짜 ISO 문자열, 금액 정수(원), enum은 `commonCodes` 참조값.
- **참조:** 깊은 임베드 금지, FK(ID) + 목록 표시용 소수 필드만 denormalize(예: `itemName`).
- **헤더-라인:** 강한 종속(주문 라인 등)만 서브컬렉션, 그 외 최상위 + FK.
- **감사:** 모든 쓰기는 `auditLogs` 기록(트리거 권장).
- **권한:** `roleGroups` → Auth custom claims → Security Rules 강제.

## 7. 구현 우선순위 (마이그레이션 순서)

[[비즈니스_로직_개발_전략.md]]의 수직 슬라이스 패턴으로 진행.

1. **공통 마스터** — `commonCodes` → `items` → `vendors` → `users`/`roleGroups` (모든 모듈 참조)
2. **생산 기반 마스터** — `boms`/`routings`/`processes`/`workCenters`/`equipments`
3. **재고 원장** — `stockMovements`+`stocks`+`lots` (정합성 핵심 먼저)
4. **트랜잭션 모듈** — 생산 → 품질 → 설비 → 자재 → 영업 (채번·상태머신·트랜잭션)
5. **집계·리포트** — `aggregates/*` (위 데이터 완성 후, 읽기전용 소비자)
6. **시스템** — `counters`(채번은 4 시작 전 필요)·`auditLogs`·권한 규칙
