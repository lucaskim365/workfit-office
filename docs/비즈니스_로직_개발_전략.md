## 1. 비즈니스 로직을 "어디에" 둘지부터 정한다 (최우선 원칙)

로직 종류에 따라 위치를 분리한다. 한 곳에 몰면 반드시 무너진다.

| 로직 종류 | 예시 | 둘 곳 |
|---|---|---|
| **순수 계산** | OEE, Cpk, 수율, 여신 소진율, 재고 수불(기초+입고−출고), FIFO | **클라이언트 순수 함수** (`domain/`) — Firebase 의존 0, 단위테스트 100% |
| **단순 CRUD** | 거래처·품목·사용자 등록/수정 | **Repository → Firestore** 직접 |
| **트랜잭션/원자성** | 실적 확정 시 재고 차감+LOT 생성, 채번(WO-260621-021) | **Cloud Functions** 또는 Firestore Transaction |
| **권한 필요/신뢰 경계** | 권한 변경, 마감, 금액 확정 | **Cloud Functions (callable)** + Security Rules |
| **무거운 집계** | 리포트 31종, 대시보드 | 데이터 적으면 클라 집계 / 크면 **사전집계 문서(aggregates)** |

> 원칙: **계산은 클라에서 순수하게, 쓰기·권한·정합성은 서버에서.** 화면은 끝까지 dumb하게 유지.

## 2. 권장 계층 구조 (수직 슬라이스)

`mock.ts`를 4계층으로 분해한다.

```
src/
  domain/                    # 순수 비즈니스 로직 (Firebase 무관, 테스트 가능)
    production/
      schema.ts              # zod 스키마 = 타입 + 검증의 단일 정의
      calc.ts                # OEE/달성률 등 순수 함수
      status.ts              # 상태머신 (대기→진행→완료/지연)
  data/                      # Firestore 접근 (Repository)
    production/
      workOrder.repo.ts      # CRUD + 쿼리
    converters.ts            # Firestore ↔ 도메인 타입 변환
  features/                  # React 바인딩 (TanStack Query 훅)
    production/
      useWorkOrders.ts       # useQuery/useMutation
  modules/...                # 기존 화면 — mock import만 훅으로 교체
```

**데이터 흐름 한 줄 요약:** `zod schema → repo(Firestore) → query hook → 화면`

### 코드 스케치 (작업지시 예)

```ts
// domain/production/schema.ts — 단일 진실 공급원
export const WorkOrder = z.object({
  id: z.string(), no: z.string(),       // WO-260621-021
  product: z.string(), line: z.string(),
  qty: z.number().int().positive(),
  done: z.number().int().nonnegative(),
  status: z.enum(['대기','진행','완료','지연']),
});
export type WorkOrder = z.infer<typeof WorkOrder>;

// domain/production/calc.ts — 순수, 테스트 대상
export const progressRate = (w: WorkOrder) => Math.round((w.done / w.qty) * 100);

// data/production/workOrder.repo.ts — Firestore
export const workOrderRepo = {
  list: (line?: string) => getDocs(query(collection(db!,'workOrders'), ...)).then(map),
  update: (id, patch) => updateDoc(doc(db!,'workOrders',id), patch),
};

// features/production/useWorkOrders.ts — React
export const useWorkOrders = (line?: string) =>
  useQuery({ queryKey: ['workOrders', line], queryFn: () => workOrderRepo.list(line) });
```

화면에서는 단 한 줄 교체:

```ts
// const rows = WO_ROWS;            // 기존 mock
const { data: rows = [] } = useWorkOrders(line);  // 실데이터
```

## 4. MES 특화 난제별 처리 전략

- **채번 (WO/LOT/전표번호):** `Date.now()` 금지. Firestore 트랜잭션 카운터 또는 Cloud Function. 동시성 충돌 핵심 포인트.
- **상태 머신:** 작업지시·검사·8D(D1~D8)·여신 등은 허용 전이를 `status.ts`에 명시(`canTransition(from, to)`). 임의 상태 변경 차단.
- **트랜잭션 정합성:** "실적 확정 → 재고 차감 + LOT 생성 + 원가 반영"처럼 여러 문서가 함께 바뀌는 건 **반드시 트랜잭션/함수**. 화면에서 순차 쓰기 금지.
- **권한(RBAC):** 정의된 role group(ADMIN/OPERATOR/FIELD_ADMIN/MT_ADMIN/MT_USER/QC_USER)을 **Auth custom claims**로 올리고 Security Rules에서 강제. UI 숨김은 보조 수단.
- **감사추적:** 이미 화면 존재(`/report/audit`). 쓰기 경로에 감사 로그 자동 기록(함수 트리거 권장).
- **추적성(LOT 계보):** 양방향 링크를 denormalize해서 읽기 최적화. `/report/trace`가 소비.
- **실시간(대시보드/라인가동):** `onSnapshot` 구독 또는 Query `refetchInterval`. 30초 자동 갱신 UI는 이미 준비됨.

## 5. 마이그레이션 경로 (저위험·점진적)

mock.ts를 한 번에 다 지우지 않는다.

1. **mock = 시드(seed)로 재활용** → Firestore 초기 데이터로 업로드.
2. **모듈 1개를 수직 슬라이스로 끝까지** (schema → repo → hook → 화면 교체 → 규칙 → 테스트).
3. 그 패턴을 나머지 모듈에 복제 (UI 포팅 때처럼 배치 진행).
4. 리포트/대시보드는 **읽기 전용이라 가장 마지막** (집계 소비자).

## 6. 권장 진행 순서 및 첫 액션

**파일럿 권장 모듈: 기준정보(거래처·품목).**
CRUD가 단순하고 다른 모든 모듈이 참조하는 마스터라 가장 먼저 실데이터화해야 효과가 크다.
생산/품질 같은 트랜잭션 모듈은 마스터가 실데이터가 된 뒤 진행해야 깔끔하다.

### 실행 옵션

- **A. 기반 스캐폴딩** — TanStack Query 설치·Provider 연결, `domain/data/features` 폴더 + 컨버터·repo 베이스, Firestore 보안규칙 초안.
- **B. 파일럿 1모듈 수직 슬라이스** — 거래처관리를 schema → repo → hook → 화면 교체까지 end-to-end로 완성해 "정본 패턴" 확립.
- **C. 데이터 모델 설계서** — 179화면의 mock을 분석해 Firestore 컬렉션/문서 스키마·관계·채번·상태머신 전체 설계 문서 작성.

> **권장 순서: C(설계) → A(기반) → B(파일럿).**
> 전체 데이터 모델을 먼저 합의해야 모듈 간 참조가 꼬이지 않는다.
