# 동적 결재선 룰 엔진(Dynamic Route Engine) 개발 계획서

> 작성일: 2026-07-07 · 대상: 그룹웨어 전자결재 Phase 6 — 기안자 위치(부서·직급) 기반 결재선 자동 구성 + 기준정보 관리 프로그램
> 상위 문서: `전자결재_워크플로_개발_계획서.md`(결재 엔진·결재선 3방식) · 관련: `데이터_모델_설계서.md`, `DB_이관_대비_설계원칙.md`, 메모리 `groupware-feature`·`data-layer-pattern`
> 정본 와이어프레임: **없음** → 기존 기준정보(마스터-디테일) 디자인 언어 재사용 ([[wireframe-source-of-truth]])

## 0. 이 문서의 범위

같은 문서 양식(예: 지출결의서)이라도 **기안자의 소속 부서·직급에 따라 결재선이 완전히 달라지는** 한국형 결재 현실을, 하드코딩 없이 **룰 엔진 + 관리자용 기준정보 프로그램**으로 해결한다.

```
[문서 상신] → [기안자 ID] → [조직도 상향 추적] → [매트릭스 룰 매칭] → [셀프결재 제외] → [동적 결재선 steps[]]
```

**두 산출물**:
1. **동적 결재선 룰 엔진** — 기안자 위치 + 문서유형 + 금액을 입력받아 결재선을 실시간 생성하는 **순수 해석기**.
2. **결재선 규칙 관리 프로그램**(기준정보) — 룰을 등록·수정·시뮬레이션하는 관리자 화면. **메뉴 위치 = 기준정보(M_BASE)**.

## 1. 배경 · 현황 (코드베이스 실측)

전자결재 Phase 2에서 **자동 상신선**을 이미 구현했으나, 단순 상급자 체인 절단 수준이라 부서·직급별 분기를 못 한다. 확장이 필요한 지점을 정확히 짚는다.

### 1.1 이미 갖춘 토대 (재사용)
| 자산 | 위치 | 역할 |
|---|---|---|
| `departments.parentId` | `domain/department/schema.ts` | **상향 추적**의 근간(부서 계층). ✔ 이미 존재 |
| `departments.headUserId` | 〃 | 부서장 — 합의/전결 라우팅 |
| `users.managerId` | `domain/user/schema.ts` | 직속 상급자(FK) — 상급자 체인 원천 |
| `useOrgTree.managerChain(uid, n)` | `features/gw/useOrgTree.ts` | managerId n단계 상향(없으면 부서장 폴백) |
| `useOrgTree.deptHeadOf(uid)` | 〃 | 소속·상위 부서장 도출(자기 자신이면 상위로) |
| `POSITION_RANK` | 〃 | 직급 서열(관리자>파트장>반장>담당>사원) — **코드 상수(하드코딩)** |
| `approvalRule{docType,amountFrom/To,finalApproverKey}` | `domain/approvalRule/schema.ts` | 전결규정(금액×유형) |
| `buildAutoLine(chainIds, rule)` | `features/gw/useAutoLine.ts` | 체인 절단 + 마지막 전결. `KEY_DEPTH` **스텁 매핑** |
| `ApprovalStep{seq,parallelGroup,kind,approverId,…}` | `domain/approvalDoc/schema.ts` | 결재선 노드(엔진 최종 산출물) |

### 1.2 현재 한계(이 계획이 채움)
1. **부서 유형 구분 없음** — `departments`에 본사/공장/영업소 구분이 없어 "공장은 공장장 전결" 같은 조직별 규칙 불가(상황 C 미해결).
2. **직급 조건 분기 없음** — 룰이 `docType×amount`만 본다. "팀장이 기안하면 본부장부터" 같은 **기안자 직급별 라인**을 정의할 수 없다(상황 B 미해결).
3. **관계형 단계가 '깊이 N'뿐** — `KEY_DEPTH`(팀장1/부서장2…)는 managerId 체인 깊이로만 근사. '공장장','재무팀장 합의' 같은 **명시적 역할 해석** 불가.
4. **규칙이 코드에 박힘** — `KEY_DEPTH`·`POSITION_RANK`가 소스 상수라 조직 개편 시 코드 수정 필요.
5. **관리 UI 부재** — 룰을 등록·수정·검증할 관리자 화면이 없다.

> 즉, **엔진의 뼈대(상향 추적·체인·부서장)는 있으나, ①부서유형 ②직급조건 ③관계형 역할 ④데이터화된 룰 ⑤관리 프로그램**이 빠졌다. 이 5가지를 신설한다.

## 2. 목표 · 비목표

### 목표
1. 기안자 위치(부서·부서유형·직급) + 문서유형 + 금액에 따라 **결재선을 데이터 룰로 동적 생성**.
2. 룰을 **기준정보에서 등록·수정·우선순위·시뮬레이션**하는 관리자 프로그램(조직 개편 시 코드 무수정).
3. 세 알고리즘 구현: **①상향 추적 ②매트릭스 룰 매칭 ③셀프 결재 방지**.
4. 데이터 계층 정본 패턴 준수 — repo만 Firestore를 알고, 엔진은 순수함수([[data-layer-pattern]]).
5. 기존 자동 상신선(`useAutoLine`)을 새 엔진(`useRouteEngine`)으로 **하위호환 대체**, 상신 모달·휴가가 그대로 소비.

### 비목표
- AI 기반 결재선 추천, 조건 스크립트(수식 엔진)까지의 범용화 — 룰은 선언형 매트릭스로 한정.
- 병렬 합의 정족수 조합(과반/만장일치) — 기존 정책(전원 합의) 유지.
- 조직 발령 이력(effective-dated) — 현재 스냅샷 기준. 이력관리는 후속.

## 3. 기안자 위치별 결재선 — 요구 시나리오 (재현 목표)

| 상황 | 기안자 위치 | 금액 | 생성 결재선 | 핵심 규칙 |
|---|---|---|---|---|
| **A** | 마케팅팀 / 사원 | 50만 | 사원(기안) → **마케팅팀장**(전결) | 부서 내 팀장 전결 |
| **B** | 마케팅팀 / 팀장 | 50만 | 팀장(기안) → **마케팅본부장**(전결) | 본인=팀장 → **자기 제외**, 상위 본부장부터 |
| **C** | 울산공장 생산팀 / 사원 | 50만 | 사원(기안) → 생산팀장(검토) → **공장장**(전결) | 부서유형=공장 → 공장장 최종 |

이 표를 §5 엔진이 **동일 룰셋으로 재현**하는 것이 완료 기준이다.

## 4. 세 알고리즘 → 우리 코드 매핑

### ① 상향 추적 (Upper-Bound Tracking)
기안자 부서/직급에서 조직도를 위로 훑어 **관계형으로** 결재자를 찾는다.
- 부서 계층: `departments.parentId` 역추적(팀→본부→공장/법인). **이미 존재**.
- 사람 체인: `users.managerId`(직속) — `useOrgTree.managerChain`. **이미 존재**.
- 부서장: `departments.headUserId` 또는 직급 seniority — `useOrgTree.deptHeadOf`. **이미 존재**.
- 신설: **부서 유형별 역할 해석**(공장장 = deptType='공장' 최상위 부서의 headUserId), **직급 기준 해석**(rank ≥ 특정 직급 최초 상급자).

### ② 매트릭스 룰 테이블 (Matrix Rule Table)
`[문서유형 × 부서범위 × 직급범위 × 금액구간]` → **표준 결재 단계(관계형)** 를 저장하는 마스터(§5.2 `approvalRouteRules`). 관리자 화면에서 CRUD. 엔진은 기안자 조건으로 **우선순위 순 첫 매칭 룰**을 고른다.

### ③ 셀프 결재 방지 (Self-Approval Block)
관계형 해석 결과 **기안자 본인이 결재선에 포함되면 자동 제외(skip)** 하고 다음 상급자로 승격. 상황 B의 "팀장이 자기 자신을 넣지 않음"이 이 로직으로 처리된다. 순수함수 `dedupeSelf`로 구현.

## 5. 데이터 모델 설계

### 5.1 조직 마스터 확장 — 부서 유형 (`departments`)
`domain/department/schema.ts`에 필드 추가(무해 마이그레이션, zod default).

| 필드 | 타입 | 설명 |
|---|---|---|
| `deptType` | enum `본사\|공장\|영업소\|연구소\|기타`(기본 `본사`) | 조직 유형 — **역할 해석·룰 매칭 키**(공장장 등) |
| `costCenter` | `string\|null` | (옵션) 비용센터 코드 — 금액 룰 확장 여지 |

- `parentId`(상향추적)·`headUserId`(부서장)는 **이미 존재** → 그대로 활용.

### 5.2 직급 마스터 신설 (`positions`) — 기준정보
현재 `users.position`은 자유문자열, 서열은 코드 상수(`POSITION_RANK`). 이를 **데이터화**한다.
`domain/position/schema.ts`

| 필드 | 타입 | 설명 |
|---|---|---|
| `id` | string(PK) | `P###` |
| `name` | string | 직급명(`users.position`과 매칭되는 표준명) |
| `rank` | number | 서열(작을수록 상위: 대표1 … 사원9) — **직급범위 비교 키** |
| `isDeptHead` | boolean | 부서 책임자 직급 여부(팀장·공장장 판정 보조) |

- `useOrgTree`의 하드코딩 `POSITION_RANK`를 이 마스터로 교체(폴백은 유지).

### 5.3 결재선 룰 테이블 신설 (`approvalRouteRules`) — 룰 엔진 핵심
`domain/approvalRoute/schema.ts`. 기존 `approvalRules`(금액×유형만)를 **포괄·대체**한다(§8 마이그레이션).

| 필드 | 타입 | 설명 |
|---|---|---|
| `id` | string(PK) | 룰 ID |
| `name` | string | 룰 이름(관리 표시) |
| `priority` | number | **우선순위**(작을수록 먼저 매칭) — 구체적 룰이 앞 |
| `active` | boolean | 사용 여부 |
| **적용 조건(match)** | | |
| `docType` | enum(동일) `\|전체` | 문서 유형(전체=모든 유형) |
| `deptScope` | object | 부서 범위(§5.4) — 특정부서/서브트리/부서유형/전체 |
| `positionFromRank`/`positionToRank` | number\|null | 기안자 **직급 범위**(rank 구간, null=무한) |
| `amountFrom`/`amountTo` | number\|null | 금액 구간(null=무한) |
| **결재 단계(steps)** | | |
| `steps` | `RouteStep[]`(임베드) | 관계형 결재 단계 정의(§5.5) |

### 5.4 `deptScope` — 부서 적용 범위
```
{ kind: '전체' }
{ kind: '부서', deptId: 'D210' }            // 특정 부서만
{ kind: '서브트리', deptId: 'D200' }         // 그 부서 + 하위 전체(parentId 재귀)
{ kind: '부서유형', deptType: '공장' }        // 유형 일치 부서 전체
```

### 5.5 `RouteStep` — 관계형 결재 단계(핵심)
사람 이름이 아니라 **기안자 기준 관계**로 저장 → 실시간 해석.

| 필드 | 타입 | 설명 |
|---|---|---|
| `resolver` | enum(아래) | 결재자 **해석 규칙** |
| `arg` | `string\|number\|null` | 해석 인자(직급 rank·부서ID·역할키 등) |
| `kind` | enum `결재\|검토\|합의\|참조\|전결` | 결재 구분(ApprovalStep.kind 매핑, 검토=결재의 표시변형) |
| `dedupeSelf` | boolean(기본 true) | 기안자면 제외하고 상위로 |
| `optional` | boolean(기본 false) | 해석 실패(체인 끊김) 시 skip 허용 |

**resolver 종류**
| resolver | arg | 해석 |
|---|---|---|
| `MANAGER` | n(기본1) | 직속 상급자 n차(`managerChain`) |
| `DEPT_HEAD` | — | 소속 부서장(`deptHeadOf`) |
| `PARENT_DEPT_HEAD` | level(기본1) | 상위 level단계 부서의 부서장(팀장→본부장→…) |
| `ROLE_FACTORY_HEAD` | — | **공장장** = 기안자 상위 부서 중 `deptType='공장'` 최상위의 headUserId |
| `ROLE_DIVISION_HEAD` | — | **본부장** = 상위 부서 중 본부급 headUserId |
| `ROLE_CEO` | — | **대표** = 최상위(parentId=null·rank 최상) |
| `POSITION_AT_LEAST` | rank | rank 이상인 **최초 상급자**(상향 탐색) |
| `SPECIFIC_USER` | userId | 고정 지정(예: 감사 참조) |
| `SPECIFIC_DEPT_HEAD` | deptId | 특정 부서장(예: 재무팀장 합의) |

- 최종(전결) 지점 = `kind='전결'` 인 노드(그 이후 절단). §4.4 엔진과 동일.

### 5.6 Firestore 적합성
- `steps`·`deptScope`는 **배열-of-맵 / 맵**이라 중첩배열 제약 무관(코덱 불필요, [[이슈_Firestore_중첩배열_코덱.md]] 교훈).
- 신규 컬렉션 `positions`·`approvalRouteRules`(+`departments.deptType`) → **firestore.rules 허용 목록·시드러너 등록**(전자결재 Phase 4와 동일 절차).

## 6. 룰 엔진 — 순수 해석기

`domain/approvalRoute/engine.ts` (UI/DB 무관 순수함수, 시각·랜덤 미사용)

```ts
resolveRoute(ctx): ApprovalStep[]
// ctx = { drafter: User, docType, amount, org: OrgTree, positions: Position[], rules: RouteRule[] }
```

**파이프라인**
1. **매칭**: `active` 룰을 `priority` 오름차순 정렬 → 기안자로 조건 평가(`docType` ∧ `deptScope` ∧ `직급 rank ∈ [from,to]` ∧ `amount ∈ [from,to]`) → **첫 일치 룰** 선택.
2. **해석**: 룰의 `RouteStep[]`을 순회하며 각 `resolver`를 org 트리로 **실인물 해석**(상향 추적).
3. **셀프 제외**: `dedupeSelf` 노드가 기안자로 해석되면 skip → 같은 resolver로 한 단계 위 승격(또는 optional이면 제거).
4. **중복 제거**: 동일 인물이 연속/중복 결재자면 상위 kind 우선으로 1회로 병합.
5. **절단**: `전결` 노드 등장 시 이후 단계 생략.
6. **폴백**: 매칭 룰이 없으면 기존 `buildAutoLine`(managerChain) 방식으로 안전 폴백.
7. 결과를 `seq` 부여해 `ApprovalStep[]`으로 반환(상신 모달이 그대로 소비, 사용자가 이어서 수정 가능).

**불변식**: 순수함수(동일 입력=동일 출력), org/positions/rules는 읽기만, 기안자는 결코 결재자에 포함되지 않음.

### 6.1 상황 A/B/C 재현(룰셋 예시)
```
[R1] 지출결의·전체부서·직급(팀장~사원 rank≥팀장)·~100만
     steps: [ DEPT_HEAD(전결) ]                       → A: 사원→팀장(전결)
[R2] 지출결의·전체부서·직급(팀장 rank=팀장)·~100만  (priority < R1)
     steps: [ PARENT_DEPT_HEAD(1)(전결) ]              → B: 팀장→본부장(전결, 자기 제외)
[R3] 지출결의·부서유형=공장·전체직급·전체금액       (priority 최상)
     steps: [ DEPT_HEAD(검토), ROLE_FACTORY_HEAD(전결) ] → C: 사원→생산팀장→공장장(전결)
```
- 우선순위: R3(공장, 가장 구체) < R2(팀장 조건) < R1(일반). 엔진이 기안자 위치로 알맞은 룰을 자동 선택.
- **셀프 제외**가 B에서 DEPT_HEAD를 자기 자신으로 해석하지 않게 하여 본부장부터 시작.

## 7. 기준정보 관리 프로그램 (위치 = 기준정보 M_BASE)

관리자가 코드 수정 없이 룰을 다루는 CRUD 화면. **메뉴 위치를 기준정보에 신설**한다.

### 7.1 메뉴 등록 (`src/app/menu-tree.ts`, `M_BASE` 하위 새 그룹)
```
G_BASE_ORG  '조직·결재 기준정보'  (order 15)
  ├ S_BASE_DEPT      부서/조직관리     /base/department     ← departments CRUD(트리 편집·deptType·headUserId)
  ├ S_BASE_POSITION  직급관리          /base/position       ← positions CRUD(rank·부서책임자)
  └ S_BASE_APRULE    결재선 규칙관리    /base/approval-rule  ← 룰 엔진 관리(핵심)
```
- 화면 컴포넌트는 `src/modules/base/<name>/`에 두고 `src/app/App.tsx` `SCREEN_COMPONENTS`에 등록(기존 기준정보 화면과 동일 패턴).
- 도크 그룹웨어의 조직도(`/gw/orgchart`, 읽기전용 뷰)와 **같은 `departments` 마스터를 공유**(도크=열람, 기준정보=편집).

### 7.2 부서/조직관리 (`/base/department`)
- 좌: 부서 트리(드래그로 parentId 변경). 우: 선택 부서 편집(이름·상위·**부서유형**·부서장 지정). 추가/삭제.
- 검증: 순환 참조(자기 조상으로 이동) 차단, 부서장 소속 확인.

### 7.3 직급관리 (`/base/position`)
- 직급 목록(rank 순) + 편집(이름·rank·부서책임자 여부). `users.position` 표준화 연동.

### 7.4 결재선 규칙관리 (`/base/approval-rule`) — 핵심 프로그램
- **목록(좌)**: 우선순위 순 룰 리스트(이름·유형·부서범위·직급범위·금액·사용여부). 드래그로 우선순위 변경.
- **편집(우)**:
  - **적용 조건**: 문서유형 / 부서범위(전체·부서·서브트리·부서유형) / 직급범위(rank from~to, 직급 마스터 드롭다운) / 금액구간.
  - **결재 단계 빌더**: `RouteStep` 행 추가 — resolver 선택(직속상급자·부서장·상위부서장·공장장·본부장·대표·직급이상·특정인·특정부서장) + 인자 + kind(결재/검토/합의/참조/전결) + 셀프제외/옵션 토글 + 순서.
  - **미리보기 시뮬레이터**: 기안자·금액 선택 → **엔진을 실제 호출**해 생성될 결재선을 즉시 표시(상황 A/B/C를 관리자가 눈으로 검증). 어떤 룰이 매칭됐는지 표시.
- **저장 즉시 반영**: 상신 모달의 자동 결재선이 새 룰을 사용(캐시 무효화).

### 7.5 상신 측 연동
- `features/gw/useAutoLine` → `features/gw/useRouteEngine`로 대체(하위호환). 상신 모달·휴가의 "자동 상신선/전결규정 적용" 버튼이 룰 엔진 결과를 프리필. 사용자는 이어서 수동 수정 가능(결재선 3방식은 유지).

## 8. 계층 설계 (정본 패턴 준수)

```
src/domain/department/schema.ts        — deptType·costCenter 필드 추가
src/domain/position/schema.ts          — 직급 마스터(신규)
src/domain/approvalRoute/schema.ts     — RouteRule·RouteStep·deptScope(신규, SSOT)
src/domain/approvalRoute/engine.ts     — resolveRoute 순수 해석기(상향추적·매칭·셀프제외)
src/data/position/position.repo.ts     — list/get/save. Firebase는 여기서만
src/data/approvalRoute/approvalRoute.repo.ts — list/get/save/remove/reorder
src/data/seeds/position.seed.ts / approvalRoute.seed.ts — 폴백+시드(상황 A/B/C 룰 포함)
src/features/gw/useOrgTree.ts          — POSITION_RANK를 positions 마스터로 교체(폴백 유지) + deptType 활용
src/features/gw/useRouteEngine.ts      — org+positions+rules → resolveRoute (useAutoLine 대체)
src/modules/base/department/DepartmentScreen.tsx — 부서/조직 CRUD
src/modules/base/position/PositionScreen.tsx     — 직급 CRUD
src/modules/base/approvalRule/ApprovalRuleScreen.tsx     — 룰 목록+편집
src/modules/base/approvalRule/RouteStepBuilder.tsx       — 결재 단계 빌더
src/modules/base/approvalRule/RoutePreview.tsx           — 시뮬레이터(엔진 호출)
src/app/menu-tree.ts + src/app/App.tsx  — G_BASE_ORG 3화면 라우트 등록
firestore.rules + scripts/seed-firestore.ts — positions·approvalRouteRules 등록·배포(+departments.deptType)
```

- **엔진 순수성**: `resolveRoute`는 `domain`의 순수함수 → 단위검증·시뮬레이터·상신 프리필이 공유.
- **마이그레이션(approvalRules→approvalRouteRules)**: 기존 `approvalRule{finalApproverKey}`를 RouteStep 1개(`PARENT_DEPT_HEAD`/`ROLE_*` 전결)로 변환하는 어댑터 제공 → 기존 시드/문서 무손상. 구 테이블은 읽기 폴백으로 한동안 병행 후 제거.
- **repo 가드**: `isFirebaseConfigured && db ? Firestore : seed`([[data-layer-pattern]]).

## 12. 확정 요청 사항

1. **부서 유형 목록**: 본사/공장/영업소/연구소/기타 — 실제 조직에 맞게 확정 필요.
