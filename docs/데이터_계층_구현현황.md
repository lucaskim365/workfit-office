# 데이터 계층 구현 현황

> 갱신: 2026-06-24
> 대상: WorkFit MES 플랫폼 (`/Users/lucaskim/Developer/Work/DemoMes_v0`)
> 선행 문서: [[비즈니스_로직_개발_전략.md]] · [[DB_이관_대비_설계원칙.md]] · [[데이터_모델_설계서.md]]

비즈니스 로직 1단계(데이터 계층). **117개 엔티티 슬라이스 + 118개 Firestore 컬렉션(시드 685건) + 5개 cross-entity 서비스**가 구현되어, **6대 핵심 업무 모듈(품질·설비·생산·자재·영업·시스템)의 데이터 계층이 완료**되었다. Firebase(Firestore)를 실제 백엔드로 도입했고, 모든 repo는 Firebase 미설정 시 seed로 graceful degrade한다.

---

## 1. 계층 구조

```
src/domain/<entity>/schema.ts   zod 단일 스키마(SSOT) + 순수 도메인 함수(계산·상태·도출)
src/domain/<entity>/status.ts   (상태머신만) canTransition / nextStatus / nextActionLabel
src/data/seeds/<e>.seed.ts      Firebase 미설정 시 폴백 + 초기 seed 소스
src/data/<entity>/<e>.repo.ts   Firestore 캡슐화 — firebase import는 여기서만
src/services/<x>.service.ts     여러 repo를 가로지르는 트랜잭션성 업무(cross-entity)
src/features/<entity>/use<E>.ts TanStack Query 훅(useQuery/useMutation)
src/modules/.../<E>Screen.tsx   화면 — 훅만 소비, firebase·repo 모름
```

- 서버상태: `@tanstack/react-query` (`src/app/QueryProvider.tsx`, staleTime 30s)
- **Firebase 가드**: 모든 repo가 `isFirebaseConfigured && db` 분기 → 미설정이면 in-memory seed로 graceful degrade. 앱이 끊김 없이 동작.
- DB 이관 시 `data/` 계층(repo)만 재작성. ([[DB_이관_대비_설계원칙.md]] 원칙 1)

## 2. Firebase 백엔드 도입 (2026-06-24)

설계가 Firestore 전제로 되어 있어 코드 대부분이 준비되어 있었고, 운영 가드를 채웠다. 상세는 메모리 `firebase-backend-setup`.

- **DB**: `gen-lang-client-0703198317`의 **named database** `ai-studio-…`(Enterprise, `(default)` 없음). 웹 SDK 설정은 `.env.local`(`VITE_FB_*`)에 채워짐.
- **보안 룰** (`firestore.rules`): 기본 전면 차단 + **본 앱 118개 컬렉션만 화이트리스트 개방**(UI 게이트). named DB에 배포 완료. `firebase.json`에 `database` 명시.
- **인증 게이트** (`src/app/auth/`): `AuthProvider`(Firebase Auth 이메일/비번)·`LoginScreen`·`AuthGate`. **`VITE_AUTH_ENABLED` 토글로 ON/OFF**(현재 OFF, 콘솔 Auth 준비 후 켜기). Firebase 미설정 시 demoMode 통과.
- **시드 러너** (`scripts/seed-firestore.ts`, `npm run seed`): firebase-admin로 룰 우회, named DB에 **118컬렉션 685건** 1회 적재. `serviceAccount.json`(gitignore) 필요. `-- --dry`로 검증.
- ⚠ 자체 로그인을 쓸 경우 룰은 `request.auth` 못 보므로 개방형 유지(화면 게이트). 진짜 보안은 Cloud Function custom token 후속 과제.

## 4. 구조 패턴 카탈로그

거의 모든 MES 데이터가 아래 패턴 중 하나(또는 조합)에 해당한다.

### ① 단순 마스터
list(filter) + save(upsert) [+ remove]. 문서ID = 자연키(code/no/lot/id). 대표: `items`, `equipments`, `spareParts`.

### ② 모달 CRUD (RHF + zod)
폼 스키마를 default 없이 분리(RHF 입출력 타입 일치). 채번 nextId, update로 시스템필드 보존. 대표: `users`.

### ③ header-line (헤더 + 임베드 라인)
항상 함께 로드되는 bounded 데이터는 헤더 문서에 라인 배열 임베드. 대표: `boms`·`routings`·`inspectionStandards`(제품→공정→항목 3중)·`materialRequests`·`inspections`(items 측정값). z.lazy 재귀 트리: `equipBoms`.

### ④ 원장 → 스냅샷
모든 수량 변동은 원장(`stockMovements`)에 1건씩, 현재고(`stocks`)는 `deriveStocks()` 순수함수로 도출. 현재고 직접수정 금지 — `repo.addMovement`만 변경.

### ⑤ 상태머신 + 채번
`domain/<e>/status.ts`의 `canTransition`/`nextStatus`로 허용 전이만. `repo.transition`이 검증. 채번은 `counters` 채널 원자 발급. **채번 포맷**(`domain/numbering.ts`): YYMMDD형(WO/NCR/CL/COA…) + **YYMM 월단위형**(PM/BM/PC/OS/SC/DS — `yymm()`+`formatDocNo()`).
대표: `workOrders`·`inspections`·`nonconformances`·`bmActions`·`subconOrders`·`spareScraps`·`liveAlarms`·`dailyChecks` 등.

### ⑥ cross-entity 서비스 (트랜잭션 정합성)
여러 repo를 가로지르는 업무를 `services/<x>.service.ts` 한 함수에 캡슐화. 운영 시 이 함수만 Firestore 트랜잭션/Cloud Function으로 원자화 → 화면·repo 무변경. feature 훅이 관련 캐시 모두 무효화.

### (변형) 조회 모니터링 마스터 / 로그 마스터
대시보드·이력 화면은 상태머신 없이 list+select. SVG 차트·집계 상수는 화면에 그대로 두고 **주 데이터 배열만** 컬렉션화(보조 데이터셋·차트 상수는 화면 상수 유지). 대표: `andonStatus`·`oeeEquipments`·`equipParams`·`spcCharts`·`downtimes` 등.

> **파생상태 vs 명시적 상태머신**: 임의 전이면 ⑤ 상태머신(저장된 status), 데이터의 함수면 도출(`salesOrders.deliveryStatus`, `stock` 정상/부족/과다). 후자는 저장 안 함 → 불일치 구조적 불가능.

## 5. cross-entity 서비스 (5종)

| 서비스 | 트리거 | 연동(한 함수 캡슐화) | 무효화 캐시 |
|---|---|---|---|
| `receiving.completeReceipt`/`receiveRemaining` | 입고 처리 | 입고수량 가산 + 재고 **입고(+qty)** | receipts, stock |
| `production.completeWorkOrder` | 작업지시 완료 | 상태전이(진행→완료) + 재고 **생산입고(+qty)** | workOrders, stock |
| `shipping.completeShipment` | 출하 완료 | 출하 전이 + 수주 **납품량 가산**(상태 도출) + 재고 **출고(−qty)** | shipments, salesOrders, stock |
| `issuing.completeIssue` | 불출 완료 | 불출 전이 + 자재별 재고 **차감(−qty)** | issues, stock |
| `quality.judgeIncomingInspection` | IQC 판정 확정 | 상태전이(검사중→판정완료) + **합격→재고입고(+qty)** / **반품·불합격→NCR 자동발행** | inspections, stock, nonconformances |

재고 원장이 생산·영업·자재·**품질** 전 흐름과 연결됨. 화면(`QualIqcResultScreen` 등)은 `useJudgeIncoming` 등 cross-entity 훅을 소비.

## 6. 공통 규약

- **zod = SSOT**: 타입·폼검증·DB파싱을 한 스키마에서. RDB 이관 시 그대로 DDL.
- **타입**: 날짜 ISO/표시 문자열, 금액·수량은 의미에 맞게 number/string(표시 포맷 보존 시 string).
- **시드 입력 타입**: 선택 필드(.default)가 많은 스키마의 seed는 `z.input<typeof schema>[]`로 선언하면 항목마다 기본값을 안 써도 됨(`inspection.seed` 적용).
- **PK 없는 데이터**: 튜플 ROWS/string[][]는 객체화하고, 자연키 없으면 결정적 합성 id(`LA-01`·`WIP-01`·`DT-01` 등) 부여.
- **TS 함정**: `db`가 `firebase.ts`에서 `let` export라 클로저 내부에서 narrowing 풀림 → 블록 내 `const fdb = db;` 캡처.
- **빈 데이터 가드 필수**: 첫 렌더는 `data=[]`(쿼리 비동기). `find(...)?.`/`arr[0]` 가드, 로딩/빈 메시지.

## 8. 신규 슬라이스 추가 레시피 (복제 체크리스트)

```
[ ] domain/<e>/schema.ts        zod 스키마 + (상태머신이면) status.ts
[ ] data/seeds/<e>.seed.ts      seed(= 화면 인라인 데이터 이전 + 타입 정규화)
[ ] data/<e>/<e>.repo.ts        list/get/save (+ create 채번 / transition 상태머신)
[ ] features/<e>/use<E>.ts      useQuery + useMutation(onSuccess invalidate)
[ ] services/<x>.service.ts     (cross-entity일 때만) 여러 repo 조합
[ ] 화면 리팩터링                인라인 const 제거 → 훅 소비 + 로딩/빈 가드 (보조/차트 상수는 화면 유지)
[ ] scripts/seed-firestore.ts   import + TABLES에 {coll, docs, id} 추가
[ ] firestore.rules             appCollection 화이트리스트에 컬렉션명 추가 → firebase deploy
[ ] npm run build               TS 클린 → npm run seed -- --dry → commit
```
