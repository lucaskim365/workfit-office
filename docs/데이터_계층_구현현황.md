# 데이터 계층 구현 현황

> 갱신: 2026-06-24
> 대상: WorkFit MES 플랫폼 (`/Users/lucaskim/Developer/Work/DemoMes_v0`)
> 선행 문서: [[비즈니스_로직_개발_전략.md]] · [[DB_이관_대비_설계원칙.md]] · [[데이터_모델_설계서.md]]

비즈니스 로직 1단계(데이터 계층) 진행 현황. **13개 엔티티 슬라이스 + 2개 cross-entity 서비스**가 구현되어, MES의 **6대 구조 패턴이 모두 코드로 확립**되었다. 이후 작업은 이 패턴의 복제.

---

## 1. 계층 구조

```
src/domain/<entity>/schema.ts   zod 단일 스키마(SSOT) + 순수 도메인 함수(계산·상태·도출)
src/data/seeds/<e>.seed.ts      Firebase 미설정 시 폴백 + 초기 seed 소스
src/data/<entity>/<e>.repo.ts   Firestore 캡슐화 — firebase import는 여기서만
src/services/<x>.service.ts     여러 repo를 가로지르는 트랜잭션성 업무(cross-entity)
src/features/<entity>/use<E>.ts TanStack Query 훅(useQuery/useMutation)
src/modules/.../<E>Screen.tsx   화면 — 훅만 소비, firebase·repo 모름
```

- 서버상태: `@tanstack/react-query` (`src/app/QueryProvider.tsx`, staleTime 30s)
- **Firebase 가드**: 모든 repo가 `isFirebaseConfigured && db` 분기 → 미설정(현재)이면 in-memory seed로 graceful degrade. 앱이 끊김 없이 동작.
- DB 이관 시 `data/` 계층(repo)만 재작성. ([[DB_이관_대비_설계원칙.md]] 원칙 1)

## 3. 6대 구조 패턴 카탈로그

거의 모든 MES 데이터가 아래 6종 중 하나(또는 조합)에 해당한다.

### ① 단순 마스터
- list(filter) + save(upsert) + remove. 문서ID = 자연키(code).
- 대표: `items`, `vendors`, `workCenters`. 신규: schema→seed→repo→hook→화면 1줄 교체.

### ② 모달 CRUD (RHF + zod)
- 폼 스키마를 **default 없이 분리**(RHF 입력/출력 타입 일치). 채번 `nextId`, `update`로 시스템필드 보존, `removeMany`.
- 대표: `users`. 폼: `userFormSchema` = `domain/user/schema.ts`, 모달이 재사용.

### ③ header-line (헤더 + 임베드 라인)
- 항상 함께 로드되는 **bounded 마스터**는 헤더 문서에 라인 배열 임베드(`boms.items`/`revisions`, `routings.steps`).
- 라인이 많거나 독립 쿼리가 필요한 트랜잭션은 서브컬렉션. 제품 목록은 헤더에서 도출(별도 컬렉션 불필요).

### ④ 원장 → 스냅샷
- **모든 수량 변동은 원장(`stockMovements`)에 1건씩**, 현재고(`stocks`)는 `deriveStocks()` 순수함수로 도출.
- 현재고 직접수정 금지 — `repo.addMovement`(원장 append)만이 재고 변경. status(정상/부족/과다)도 도출.
- 운영 시 트리거가 stocks 유지하되 로직 동일.

### ⑤ 상태머신 + 채번
- `domain/<e>/status.ts`의 `canTransition`/`nextStatus`로 **허용 전이만** 가능. `repo.transition`이 검증(불가 전이 예외).
- 채번: `counters` 채널에서 시퀀스 원자 발급(`Date.now()`를 ID로 쓰지 않음).
- 대표: `workOrders`(대기→발행→진행→완료), `shipments`(출고대기→피킹중→출고완료).

### ⑥ cross-entity 서비스 (트랜잭션 정합성)
- 여러 repo를 가로지르는 업무를 `services/<x>.service.ts` 한 함수에 캡슐화. 운영 시 이 함수만 Firestore 트랜잭션/Cloud Function으로 원자화 → 화면·repo 무변경.
- feature 훅이 관련 캐시 모두 무효화.

> **파생상태 vs 명시적 상태머신**: 임의 전이면 ⑤ 상태머신(저장된 status), **데이터의 함수면 도출**(`salesOrders` 납품상태 `deliveryStatus`, `stock` 정상/부족/과다). 후자는 저장 안 함 → 데이터와 상태 불일치가 구조적으로 불가능.

## 4. cross-entity 서비스 (2종)

| 서비스 | 트리거 | 연동(한 함수에 캡슐화) | 무효화 캐시 |
|---|---|---|---|
| `production.service.completeWorkOrder` | 작업지시 완료 | 상태전이(진행→완료) + 재고 **생산입고(+qty)** | workOrders, stock |
| `shipping.service.completeShipment` | 출하 완료 | 출하 상태전이(피킹중→출고완료) + 수주 **납품량 가산**(납품상태 도출) + 재고 **출고(−qty)** | shipments, salesOrders, stock |

검증된 연쇄: 출하 `DO-2606-122` 출고완료 → 수주 `SO-2606-088` 미납→부분납품 → 재고 MX-200 −12. (3화면 자동 반영)

## 5. 공통 규약

- **zod = SSOT**: 타입·폼검증·DB파싱을 한 스키마에서. RDB 이관 시 그대로 DDL.
- **타입**: 날짜 ISO 문자열, 금액 정수(원), 수량 정수. (`safety:"500 EA"`→`safetyStock:500` 식 정규화 적용)
- **TS 함정**: `db`가 `firebase.ts`에서 `let` export라 클로저(.map 등) 내부에서 narrowing이 풀림 → 블록 내 `const fdb = db;` 캡처 후 사용(`counter.repo`, `user.repo` 적용).

## 7. 신규 슬라이스 추가 레시피 (복제 체크리스트)

```
[ ] domain/<e>/schema.ts        zod 스키마 + (필요 시) status.ts / 도출 함수
[ ] data/seeds/<e>.seed.ts      seed(= 화면 인라인 데이터 이전 + 타입 정규화)
[ ] data/<e>/<e>.repo.ts        list/save/remove (+ create 채번 / transition 상태머신)
[ ] features/<e>/use<E>.ts      useQuery + useMutation(onSuccess invalidate)
[ ] services/<x>.service.ts     (cross-entity일 때만) 여러 repo 조합
[ ] 화면 리팩터링                인라인 const 제거 → 훅 소비 + 로딩/빈 가드
[ ] npm run build               TS 클린 확인 → commit/push
```

> 빈 데이터 가드 필수: 첫 렌더는 `data=[]`(쿼리 비동기). `Math.max(...[])`=-Infinity, `arr[0]` undefined, `find(...)?.` 등 항상 가드.
