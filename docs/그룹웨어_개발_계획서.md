# 그룹웨어(Groupware) 개발 계획서

> 작성일: 2026-07-06 · 대상: 퀵 도크 우측 슬라이드 패널의 "그룹웨어" 포털 + 하위 12개 앱
> 관련 문서: `메신저_개발_계획서.md`(직전 도크 기능 정본), `데이터_모델_설계서.md`, `DB_이관_대비_설계원칙.md`, 메모리 `data-layer-pattern`
> 정본 와이어프레임: `mes layout/mes/quick-dock.jsx` → **도크 런처 그리드만 존재.** 12개 앱 각각의 화면 정본은 **없음**(신규 설계 영역)

---

## 1. 배경 · 현황

퀵 도크(`src/app/shell/QuickDock.tsx`)의 3종 도구(그룹웨어·위디·메신저) 중 메신저는 실기능 개발이 끝났고([[메신저_개발_계획서.md]]), 이번엔 **그룹웨어**를 다룬다.

**현재 상태 = 정적 목업 포털.** `GroupwarePanel`(`QuickDock.tsx:123-201`)은:
- **앱 타일 12개**(`apps` 배열)·**결재 대기 3건**(`approvals`)·**공지 3건**(`notices`)이 전부 **하드코딩**.
- 타일·카드에 **`onClick`이 전혀 없다** → 클릭해도 아무 동작 없음.
- 프로필 헤더가 **`홍채원 · 개발팀` 고정**(`profilePhoto` 자산) — 로그인 사용자와 무관.
- 배지(전자결재 3, 자원예약 99+, 문서관리 9)는 장식용 정적 숫자.

**정본 와이어프레임 확인 결과**(`quick-dock.jsx`): 그룹웨어는 **도크 런처 그리드(앱 타일 + 결재/공지 요약 카드)만** 정의돼 있다. → **12개 앱의 실제 화면(전자결재함·메일함·캘린더 등)은 와이어프레임에 없다.** [[wireframe-source-of-truth]] 원칙에 따라, 이 앱들은 **신규 설계 영역**이며 진행 범위·깊이를 §4·§10에서 확정한다.

**아키텍처 현황**: 앱 전체가 `domain → data(repo) → features(React Query) → modules` 정본 패턴([[data-layer-pattern]])이며 118개 컬렉션을 운용한다. 그룹웨어 신규 앱도 **동일 패턴**으로 붙인다(메신저에서 검증된 도크-기능 개발 흐름 재사용).

## 2. 목표 · 비목표

### 목표
1. 도크 그룹웨어를 **살아있는 포털**로: 타일 클릭 → 해당 앱 진입, 요약 위젯(결재 대기·공지·오늘 일정)이 **실데이터**로 갱신, 프로필 헤더가 **로그인 사용자**를 반영.
2. 12개 앱을 **데모 가치·재사용성 기준으로 티어링**(§4)하여, 핵심 앱은 실동작·주변 앱은 경량/차기로 **현실적으로 범위를 나눈다.**
3. 데이터 계층 정본 패턴 준수 — **repo만이 Firebase를 알고**, 화면·도메인은 무관([[DB_이관_대비_설계원칙.md]]). Firebase 미설정 시 seed로 graceful degrade.
4. 기존 도메인 최대 재사용 — **조직도·인명관리 = `users`+`companySite` 재사용**, 휴가 = 전자결재 흐름 재사용, 문서관리 = 메신저 Storage 첨부 패턴 재사용.

### 비목표(이번 범위 제외)
- 실제 메일 송수신(SMTP/IMAP), 외부 캘린더 동기화(CalDAV), 전자결재 법적 전자서명·PKI.
- 12개 앱 전부의 풀 프로덕션 구현 — 그룹웨어는 각 앱이 독립 제품급이라 **데모 범위로 계층화**한다(§4).
- 실시간 알림 푸시(도크 배지는 폴링/무효화로 갱신, 메신저와 동일 정책).

## 3. 핵심 설계 결정 — 앱 진입 방식

도크 폭은 **384px**로 좁다. 전자결재·메일·캘린더처럼 넓은 화면이 필요한 앱을 도크 안에 담으면 사용성이 무너진다. 세 가지 안:

| 안 | 방식 | 장점 | 단점 |
|---|---|---|---|
| A | **도크 = 포털 런처, 앱 = 본문 전체화면 라우트**(`/gw/<app>`) | 기존 MES 모듈과 동일 패턴, 넓은 작업공간, 라우팅·권한 재사용 | 도크가 닫히고 본문 전환(맥락 이동) |
| B | 앱도 전부 도크 슬라이드 패널 내부 | 도크 안에서 완결 | 384px에 결재/메일/캘린더 부적합, 신규 패널 수명주기 다수 |
| C | 하이브리드 | 유연 | 앱마다 진입 규칙 달라 혼란 |

### 확정(권장): **Option A — 도크는 포털, 앱은 본문 라우트**
- **도크 GroupwarePanel = 포털 홈**: ① 앱 12타일 런처 ② at-a-glance 위젯(결재 대기·공지·오늘 일정 요약). 타일 클릭 → `navigate('/gw/<app>')` + 도크 닫기.
- **각 앱 = `src/modules/gw/<app>/` 전체화면**(기존 8대 모듈과 동일한 라우트·레이아웃). `App.tsx`에 `/gw/*` 라우트 등록, 필요 시 `menu-tree.ts`에 `M_GW` 노드 추가(선택).
- 도크는 "빠른 진입 + 요약"만 담당 → 좁은 폭에 최적, 무거운 앱은 넓은 본문에서 동작.

## 4. 12개 앱 티어링 (데모 현실성 · 재사용 기준)

각 앱을 **데모 가치**(도크가 이미 노출하는가)·**재사용 가능 도메인**·**공수**로 분류한다. 사용자 제공 설명을 각 앱 범위에 반영.

| # | 앱 | 재사용 자원 | 신규 도메인 | 티어 | 데모 범위 |
|---|---|---|---|---|---|
| 1 | **전자결재**(3) | 없음(신규 핵심) | `approvalDoc` | **T1** | 결재함(대기/상신/완료) + 문서 상세 + 승인/반려. 도크 "결재 대기" 위젯 원천 |
| 2 | **게시판**(공지) | 없음 | `boardPost` | **T1** | 전사/부서 게시글 목록·상세·작성. 도크 "공지사항" 위젯 원천 |
| 3 | **조직도** | ✅ `users`+`companySite` | — | **T1** | 부서 트리 + 사원 배치도. 기존 데이터 재사용 → 저비용 |
| 4 | **인명관리** | ✅ `users`(+외부연락처) | `contact`(외부만) | **T1** | 임직원 주소록(검색) + 외부 거래처 연락처 |
| 5 | **일정관리** | 패턴참고 `scheduleEntry` | `gwEvent` | **T2** | 월간 캘린더 + 일정 CRUD. 도크 "오늘 일정" 위젯 원천 |
| 6 | **자원예약**(99+) | 없음 | `resource`+`reservation` | **T2** | 회의실/차량/장비 예약 목록·신청 |
| 7 | **휴가관리** | ✅ 전자결재 흐름 | `leaveRequest`+잔여 | **T2** | 잔여 조회 + 휴가 신청(=결재 유형) + 내역 |
| 8 | **업무관리** | 패턴참고 | `task` | **T2** | 개인/팀 TO-DO·진행 트래킹 |
| 9 | **문서관리**(9) | ✅ 메신저 Storage 첨부 | `gwDocument` | **T3** | 파일 메타 목록 + 업로드/다운로드(첨부 패턴 재사용) |
| 10 | **전자설문** | 없음 | `survey`+`response` | **T3** | 설문 목록 + 응답 폼(간이 통계) |
| 11 | **커뮤니티** | 게시판 패턴 | `communityPost` | **T3** | 게시판 변형(소모임 글). board 재사용 |
| 12 | **메일** | 없음 | `mailMessage` | **T3(최후순위)** | 경량 받은편지함 리스트(송수신 목업). 풀 클라이언트 제외 — **개발 순서 맨 마지막** |

- **T1(4)** = 도크 위젯이 직접 노출(결재·공지)하거나 기존 도메인 재사용(조직도·인명)으로 **고효과·저리스크**. → **1차 개발 대상.**
- **T2(4)** = 신규 도메인이나 **단일 리스트/캘린더**로 데모 성립. 휴가는 전자결재 흐름 재사용.
- **T3(4)** = 독립 제품급 또는 후순위. **"준비 중" 랜딩 또는 최소 리스트**로 시연, 정식 구현은 차기. **메일은 SMTP급 풀 클라이언트라 가치 대비 공수가 커 전 앱 중 개발 순서 맨 마지막.**

> **권장 진행**: **T1 → T2 → T3 순차**, T3 내에서도 **메일을 최후순위**(문서관리 → 전자설문 → 커뮤니티 → 메일). T3는 스텁 후 필요 시 승격.

## 5. 데이터 모델 (신규 도메인)

정본 카탈로그([[데이터_모델_설계서.md]])에 그룹웨어 도메인을 신설한다. **네이밍 규칙: `gw*` 프리픽스 또는 명확한 단수 엔티티명.** 아래는 T1 우선 + T2 골격.

### 5.1 `approvalDocs` — 전자결재 문서 (T1)
`src/domain/approvalDoc/schema.ts`

| 필드 | 타입 | 설명 |
|---|---|---|
| `id` | string(PK) | 문서번호(`AP-yyyy-####`, 채번) |
| `docType` | enum `기안\|품의\|지출결의\|휴가` | 문서 종류(휴가관리와 공유) |
| `title` | string | 제목 |
| `drafterId` | string | 기안자 `users.id` |
| `status` | enum `대기\|진행\|완료\|반려` | 결재 상태(상태머신) |
| `approvers` | `{ userId, order, decision, at }[]` | 결재선(순차 승인). `order`로 순서, `decision` 미결/승인/반려 |
| `body` | string | 본문 |
| `amount` | number \| null | 금액(지출결의) |
| `createdAt` | string(ISO) | 상신 시각 |

- **도크 "결재 대기" 위젯** = 현재 로그인 사용자가 다음 결재자(`approvers`에서 미결·차례)인 문서 수 도출.
- 상태 전이: 상신→대기, 각 결재자 승인 시 `진행`, 마지막 승인 `완료`, 누구든 반려 시 `반려`. ([[data-layer-pattern]] 상태머신 패턴)
- **결재선 방식 = 수동 + 자동 병행(확정)**:
  - **수동**: 상신 시 결재자 피커에서 사람을 **순차 지정**(메신저 `MemberPicker` 재사용). 스키마 변경 없음.
  - **자동(상신선)**: 기안자의 **상급자 체인**을 따라 결재선 자동 생성 → 상신 시 프리필(수정 가능). 이를 위해 **`users`에 `managerId`(직속 상급자) 필드 신설**(§5.3). 직급 서열 필요 시 `positionRank`(number) 추가 고려.
  - UI: 상신 화면에서 "자동 상신선" 버튼(상급자 체인 프리필) ↔ 수동 편집(추가/삭제/순서변경) 토글.

### 5.2 `boardPosts` — 게시판/공지 (T1)
`src/domain/boardPost/schema.ts`

| 필드 | 타입 | 설명 |
|---|---|---|
| `id` | string(PK) | 게시글 ID |
| `board` | enum `공지\|경조사\|규정\|자유` | 게시판 구분 |
| `title` / `body` | string | 제목·본문 |
| `authorId` | string | 작성자 `users.id` |
| `pinned` | boolean | 상단 고정(공지) |
| `createdAt` | string(ISO) | 작성 시각 |

- **도크 "공지사항" 위젯** = `board:'공지'` 최신 N건. **커뮤니티(T3)는 같은 스키마의 `board` 확장**으로 재사용.

### 5.3 조직도 — 기존 도메인 재사용 + 상급자 필드 신설 (1차)
- **조직도** = `users`(부서·직책) + `companySite`(사업장/부서) → **트리 도출**(파생 뷰, 신규 도메인 없음).
- **`users`에 `managerId` 신설**(직속 상급자, `users.id` FK, nullable): ① 조직도의 **보고 계층(트리)** 표현 ② 전자결재 **자동 상신선**의 상급자 체인 원천. → 조직도를 먼저 만들며 이 필드·시드를 확정하면, 전자결재 자동 결재선이 그대로 소비.
- **공통 조직 데이터 훅**(`useOrgTree`/부서별 사용자·상급자 체인)을 조직도에서 뽑아 → 조직도 화면·전자결재 결재선 피커가 **공유**.
- **인명관리**(1차 후순위) = `users` 주소록 뷰 + **외부 연락처만** 신규 `contacts`. 임직원 파트는 `users` 재사용.

### 5.4 T2 골격(요약)
- `gwEvents`(일정): `{ id, title, start, end, allDay, ownerId, category }` — 월 캘린더.
- `resources`+`reservations`(자원예약): 자원 마스터 + 예약 로그(시간대 충돌 검사).
- `leaveRequests`(휴가): `approvalDocs.docType='휴가'` **재사용** + 사용자별 잔여일수 도출.
- `tasks`(업무): `{ id, title, assigneeId, status(할일\|진행\|완료), due }`.

> **미읽음·대기·잔여 등 배지 값은 전부 도출**(저장 안 함) — 데이터-상태 불일치 불가([[data-layer-pattern]] 파생상태 원칙).

## 6. 계층 설계 (정본 패턴 준수)

메신저와 동일한 도크-기능 개발 흐름. **T1 기준** 예시:

```
src/domain/approvalDoc/schema.ts       — 결재 문서 zod SSOT
src/domain/boardPost/schema.ts         — 게시글 zod SSOT
src/domain/contact/schema.ts           — 외부 연락처 zod SSOT
src/data/approvalDoc/approvalDoc.repo.ts  — list/get/create/approve/reject. Firebase는 여기서만
src/data/boardPost/boardPost.repo.ts      — list/get/create
src/data/contact/contact.repo.ts          — list/get/save
src/data/seeds/approvalDoc.seed.ts / boardPost.seed.ts / contact.seed.ts — 폴백+시드
src/features/gw/useApprovals.ts        — 결재함 useQuery + useApprove/useReject(useMutation)
src/features/gw/useBoard.ts            — 게시글 useQuery + useCreatePost
src/features/gw/useOrgChart.ts         — users+companySite 조합 트리 도출
src/features/gw/useGwSummary.ts        — 도크 위젯용(결재대기·공지·오늘일정) 집계
src/modules/gw/approval/ApprovalScreen.tsx   — 결재함 전체화면
src/modules/gw/board/BoardScreen.tsx         — 게시판 전체화면
src/modules/gw/orgchart/OrgChartScreen.tsx   — 조직도
src/modules/gw/contacts/ContactsScreen.tsx   — 인명관리
src/modules/gw/_gw.tsx                        — 그룹웨어 공통 UI(헤더·카드·필터, _mat/_sales 패턴)
src/app/shell/QuickDock.tsx            — GroupwarePanel 목업 제거 → 포털(런처+위젯) 실동작화
src/app/App.tsx                        — /gw/* 라우트 등록
```

- **repo 가드**: `isFirebaseConfigured && db ? Firestore : in-memory seed`. `db`는 `let` export → 클로저 진입 전 `const fdb = db;` 캡처(기존 함정 회피, [[data-layer-pattern]]).
- **화면·domain·features는 firebase 무지.** 저장소 교체 시 repo만 재작성.
- **인프라 통합 파일**(메인이 직접): `firestore.rules`에 그룹웨어 컬렉션 규칙, `firestore.indexes.json`에 정렬 인덱스(결재 `createdAt`·게시 `pinned,createdAt`), `scripts/seed-firestore.ts` 시드러너에 신규 컬렉션 균일 매핑 추가.
- **공통 UI 재사용**: `_mat.tsx`/`_sales.tsx`의 `C·MHead·MKpis·FBar/FField/FSel·th/td`를 `_gw.tsx`가 재export해 화면 톤 통일.

## 7. UI 설계

### 7.1 도크 포털(GroupwarePanel 개선)
- **프로필 헤더**: 하드코딩 `홍채원` → `useAuth()` 로그인 사용자(이름·부서·아바타). 알림 배지도 도출값.
- **앱 12타일**: 각 타일에 `onClick → navigate('/gw/<app>')` + 도크 닫기. 배지(결재 3·문서 9 등)를 `useGwSummary()` 도출값으로 교체. T3 앱 타일은 "준비 중" 표식.
- **요약 위젯**: 기존 "결재 대기"·"공지사항" 카드를 실데이터화(+ "오늘 일정" 추가 가능). 각 항목 클릭 → 해당 앱 딥링크.

### 7.2 대표 앱 화면(T1, 신규 설계 — 기존 디자인 언어 준수)
- **전자결재**: 좌측 함(대기/상신/완료/반려) 탭 + 문서 리스트 + 우측 상세(결재선 진행 표시, 승인/반려 버튼). `_gw` 카드/테이블 재사용.
- **게시판**: 게시판 구분 탭 + 목록(고정글 상단) + 상세/작성 모달.
- **조직도**: 부서 트리(좌) + 선택 부서 사원 카드 그리드(우). `users`·`companySite` 재사용.
- **인명관리**: 검색 + 임직원/외부 탭 + 연락처 카드. → 와이어프레임 부재이므로 **기존 마스터-디테일/리스트 디자인 언어로 신규 구성**([[wireframe-source-of-truth]] 규칙: 정본 없을 때 기존 톤 준수 + 사용자 확인).

### 7.3 라우팅
- `App.tsx`에 `/gw/approval`·`/gw/board`·`/gw/orgchart`·`/gw/contacts`(+T2 경로) 등록.
- 선택: `menu-tree.ts`에 `M_GW` 모듈 노드 추가해 좌측 메뉴에서도 진입(도크-우선이면 생략 가능).

## 8. 단계별 로드맵

**1차 = 전자결재 워크플로 클러스터**(조직도 → 전자결재 → 휴가관리)로 재편(확정). 조직 데이터 계층을 먼저 세우고, 그 위에 결재 엔진, 다시 그 위에 휴가(=결재 유형)를 얹는다.

| Phase | 산출물 | 완료 기준(DoD) |
|---|---|---|
| **0. 포털 실동작화** | GroupwarePanel 목업 제거, 프로필=로그인 사용자, 타일 onClick→라우트, `useGwSummary` 골격(빈/seed) | 타일 클릭 시 본문 라우트 이동·도크 닫힘, 프로필 실사용자, tsc·build 통과 |
| **1. 조직도 + 조직 데이터 계층** | `users.managerId` 신설·시드, `useOrgTree`/부서별·상급자체인 훅, 조직도 화면(부서 트리+사원 배치) | 부서 트리·보고 계층 표시, 상급자 체인 도출, tsc·build 통과 |
| **2. 전자결재** | `approvalDoc` domain/repo/seed/features, 결재함(대기/상신/완료/반려)+상세, **결재선 수동+자동**(상급자체인 프리필), 도크 "결재 대기" 위젯 실데이터 | 상신→결재선→순차 승인/반려→상태전이, 자동 상신선 프리필+수동 편집, 대기 배지 도출 정확 |
| **3. 휴가관리** | `leaveRequest`(=`approvalDoc docType:'휴가'` 재사용)+잔여 도출, 휴가 신청→결재 상신, 내역 | 휴가 신청이 결재 엔진으로 흐름, 잔여일수 도출, 승인 시 완료 반영 |
| **4. 영속화** | 1~3 repo Firestore 연결, rules/index/시드러너 통합 | 새로고침·다계정에서 결재·휴가 유지, 배지 정확 |
| **5. 일정관리(다음)** | `gwEvent` 캘린더 + **승인된 휴가 자동 표시**(휴가 통합) | 월 캘린더 CRUD, 승인 휴가가 일정에 노출 |
| 이후 | 게시판·인명관리(T1 잔여) → 자원예약·업무(T2) → 문서관리·설문·커뮤니티 → **메일(최후순위)** | 티어·순서 §4 |

- Phase 순차(포털→조직데이터→결재→휴가). 인프라 3파일(`rules`·`indexes`·`seed-firestore.ts`)은 Phase 4에서 메인이 통합.
- **일정관리는 Phase 5**(1차 클러스터 밖) — 휴가 승인 결과를 캘린더로 시각화하는 통합 계층으로 합류.
- 검증: 각 Phase 후 `/run` 또는 로컬 구동으로 도크·라우트 육안 확인(메신저 검증 흐름 동일).
