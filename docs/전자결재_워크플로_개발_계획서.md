## 0. 이 문서의 범위

그룹웨어 12앱 중 **결재 워크플로로 강결합된 3종**만 독립 계획한다. 나머지(게시판·인명·일정·자원예약·업무·문서·설문·커뮤니티·메일)는 상위 계획서 소관.

```
조직도 ──(조직 데이터·상급자 체인·부서장)──▶ 전자결재 ──(docType='휴가' 재사용)──▶ 휴가관리
   Phase 1                                     Phase 2                         Phase 3
```

**설계 사상**: 조직 데이터 계층을 먼저 세우고(Phase 1) → 그 위에 범용 결재 엔진(Phase 2) → 다시 그 위에 휴가(첫 결재 적용 사례, Phase 3). 각 층이 아래 층을 **재사용**하므로 중복 도메인이 없다.

## 1. 배경 · 현황

- 도크 그룹웨어(`QuickDock.tsx` `GroupwarePanel`)는 정적 목업 — 전자결재 타일 배지 `3`·"결재 대기 2" 카드가 하드코딩, onClick 없음.
- **재사용 자산**(코드베이스 실측):
  - `users`(`src/domain/user/schema.ts`): `id·empNo·name·dept·position·roleGroup·email·status`. **부서 마스터·직급 서열·상급자 관계는 없음**(dept·position은 자유문자열).
  - `companySite`: **사업장** 마스터(부서 아님) — 조직 트리에 직접 쓸 수 없음.
  - `MemberPicker`(`QuickDock.tsx:679`, `{exclude, selected, onToggle}`): 조직/사용자 다중 선택 — **결재자 피커로 재사용**.
  - 채번 `formatDocNo(prefix, dateKey, seq)`(`domain/numbering.ts`) → `AP-YYMMDD-NNN`.
  - header-line 임베드 패턴(boms/routings): **결재선을 문서에 임베드**하는 근거.
- **핵심 공백 2가지**(이 계획이 채움): ① 조직 계층(부서 트리·상급자) 데이터가 없어 자동 결재선 불가 → Phase 1이 신설. ② 범용 결재 엔진 부재 → Phase 2가 신설.

## 2. 목표 · 비목표

### 목표
1. **조직도**: 부서 트리 + 사원 배치 + **상급자 체인/부서장**을 데이터화해, 전자결재·휴가가 소비할 **조직 데이터 계층**을 확립.
2. **전자결재**: 임시저장→상신→(순차/병렬)결재→승인/반려/회수→완료의 **범용 결재 엔진**. **결재선 3방식(수동·자동상신선·전결규정)**과 **결재구분 5종(결재·합의·참조·전결·대결)**을 지원(§4가 이 문서의 핵심).
3. **휴가관리**: 휴가신청을 `docType='휴가'` 결재 문서로 흘려 **결재 엔진을 재사용**하고, 사용자별 **잔여일수를 도출**.
4. 데이터 계층 정본 패턴 준수 — repo만 Firebase를 알고, Firebase 미설정 시 seed로 graceful degrade.

### 비목표
- 법적 전자서명(PKI)·공인인증, 결재 문서 PDF 출력/직인, 외부 ERP 연동.
- 병렬 합의의 복잡한 정족수(과반/만장일치 조합) — 병렬은 "전원 합의" 단일 규칙만(추후 확장 여지).
- 실시간 푸시 알림(도크 대기 배지는 폴링/무효화로 갱신, 메신저와 동일 정책).

## 3. 조직 데이터 계층 (Phase 1의 산출, 결재의 토대)

전자결재 자동 결재선·전결·합의 라우팅이 성립하려면 "누가 누구의 상급자인가 / 어느 부서의 장인가"가 데이터로 있어야 한다. `users`엔 없으므로 **최소 침습으로 신설**한다.

### 3.1 `users`에 `managerId` 추가 (직속 상급자)
`src/domain/user/schema.ts`에 필드 1개 추가.

| 필드 | 타입 | 설명 |
|---|---|---|
| `managerId` | `string \| null`(기본 null) | 직속 상급자 `users.id`(FK). **자동 상신선의 상급자 체인 원천** |

- 기존 시드·사용자 폼에 상급자 선택(선택 입력) 추가. `userFormSchema`에도 optional 반영.
- 마이그레이션 무해: 기존 문서에 없으면 `null`로 파싱(zod default). 화면·다른 도메인 무영향([[DB_이관_대비_설계원칙.md]]).

### 3.2 `departments` — 부서 마스터 (신규, 경량)
`src/domain/department/schema.ts`

| 필드 | 타입 | 설명 |
|---|---|---|
| `id` | string(PK) | 부서 ID(`D001`…) |
| `name` | string | 부서명(= `users.dept`와 매칭되는 표준명) |
| `parentId` | `string \| null` | 상위 부서(FK) → **트리 구성** |
| `headUserId` | `string \| null` | 부서장 `users.id` → **합의/전결 라우팅** |
| `order` | number | 형제 정렬 |

- 조직도 트리 = `departments`(parentId로 계층) × `users`(dept로 소속). 기존 `users.dept` 문자열을 표준 부서명으로 정렬.
- **부서장(headUserId)**은 합의(부서 대표 동의)·전결(부서장 전결) 라우팅에 사용.
- 시드: 현 users seed의 dept 집합(시스템관리팀·생산1팀·생산2팀·품질보증팀·현장관리팀·설비보전팀…)을 부서로 등재 + 상위(생산본부·품질본부 등) 부여.

### 3.3 조직 데이터 훅 `useOrgTree`
`src/features/gw/useOrgTree.ts` — 파생 뷰(도출), 신규 저장 없음.
- `tree`: `departments`+`users` → 중첩 트리(부서→사원).
- `managerChain(userId, n)`: `managerId`를 n단계 따라 올라간 상급자 배열 → **자동 상신선 원천**.
- `deptHead(userId)`: 소속 부서(및 상위)의 `headUserId` → **합의/전결 대상**.
- 조직도 화면과 전자결재 결재선 빌더가 **공유**.

## 4. 전자결재 — 결재 방법 (이 계획서의 핵심)

한국형 전자결재의 실무 개념을 데이터·상태·UI로 온전히 담는다.

### 4.1 결재선 구성 방식 (3가지, 병행 지원)

| 방식 | 동작 | 데이터 원천 | UX |
|---|---|---|---|
| **① 수동 지정** | 기안자가 결재자를 **직접 순차 선택** | `MemberPicker` 재사용 | 상신 화면에서 사람 추가/삭제/순서변경, 각 노드 결재구분 지정 |
| **② 자동 상신선** | 기안자 **상급자 체인** 따라 N단계 프리필 | `useOrgTree.managerChain` | "자동 상신선" 버튼 → 팀장→부서장→… 프리필 후 수정 가능 |
| **③ 전결규정 기반** | 문서유형·금액 구간별 **전결권자까지** 자동 구성/절단 | `approvalRules`(§5.3) | 유형·금액 입력 시 결재선 자동 재계산(전결권자 이하 생략/지정) |

- 세 방식은 **배타적이지 않다**: ②로 프리필 → ③ 전결규정으로 절단 → ①로 미세조정, 순으로 합성 가능.
- 최종 결재선은 어느 방식으로 만들었든 **동일한 `steps[]`(§5.2)로 저장** → 엔진은 방식과 무관하게 동작.

### 4.2 결재 구분 (각 결재선 노드의 역할, 5종)

| 구분 | 의미 | 진행 영향 |
|---|---|---|
| **결재(승인)** | 필수 순차 승인자 | 승인해야 다음 단계로 |
| **합의** | 관련 부서 **동의**(부서장 등) | 병렬 가능, 전원 합의해야 다음 단계 |
| **참조** | **열람만**(CC), 승인 불필요 | 진행 막지 않음, 알림/열람표시만 |
| **전결** | 위임된 **최종 결재** | 이 노드 승인 시 **상위 단계 생략하고 완료** |
| **대결** | 부재중 결재자를 **대신 결재** | 원 결재자 권한으로 대리 승인(위임 기록) |

- 노드 스키마의 `kind` enum으로 표현(§5.2). 참조는 `decision` 무관(항상 통과), 전결은 승인 시 잔여 결재 단계 skip.

### 4.3 결재 진행 방식

- **순차(직렬)**: `seq` 오름차순으로 한 명씩. 기본.
- **병렬(합의)**: 같은 `parallelGroup` 태그를 가진 노드들은 **동시 활성** → 전원 결정해야 그룹 통과(합의에 주로 사용).

### 4.4 결재 액션 · 상태 머신

**문서 상태**: `임시저장 · 진행중 · 반려 · 완료 · 회수`

```
임시저장 ──상신──▶ 진행중 ──(중간 단계 승인)──▶ 진행중
                     │                         │
                     ├──최종 승인/전결 승인──▶ 완료
                     ├──반려(사유 필수)──────▶ 반려 ──재상신──▶ 진행중
                     └──회수(첫 결재 전만)───▶ 회수 ──재상신──▶ 진행중
```

| 액션 | 주체 | 조건 | 결과 |
|---|---|---|---|
| **임시저장** | 기안자 | — | `임시저장` 유지(상신 전 편집) |
| **상신** | 기안자 | 결재선 1명 이상 | `진행중`, 첫 결재 단계 활성, 대기 배지 발생 |
| **승인** | 현재 단계 결재자 | 본인 차례 | 다음 단계 활성 / 마지막이면 `완료` |
| **전결 승인** | 전결권자 | `kind='전결'` | 상위 잔여 단계 생략 → `완료` |
| **반려** | 임의 결재자 | 사유(comment) 필수 | `반려`, 기안자에게 회송 |
| **회수** | 기안자 | **아직 아무도 승인 안 함** | `회수`(상신 취소) |
| **보류** | 현재 결재자 | — | 단계 `보류` 표시(진행 일시정지, 재개 가능) |
| **대결/위임** | 대결자 | 원 결재자 부재 지정 | 원 결재자 권한으로 승인, `delegatedFromId` 기록 |
| **재상신** | 기안자 | `반려\|회수` 상태 | 수정 후 다시 `진행중` |

- **불변식**: `완료`·`반려`·`회수`는 종결 상태(재상신만 예외적 전이). 단계 결정은 **오직 본인 차례(활성 단계)**에서만.
- 상태·단계 결정은 **repo의 트랜잭션적 갱신**(문서 1건 내 `steps[]`+`status` 동시 갱신) — header-line 임베드라 원자적.

### 4.5 도크 · 알림 연동 (도출)
- **도크 "결재 대기" 배지** = 로그인 사용자가 **현재 활성 단계의 결재자**인 문서 수(참조·미래 단계 제외). 저장 안 함, 도출.
- 상신/승인/반려 시 관련자에게 알림(메신저 시스템 메시지 재사용 여지 — 이번엔 도크 배지·"내가 상신한 문서" 상태 표시까지).

## 6. 계층 설계 (정본 패턴 준수)

## 7. UI 설계 (신규 — 기존 마스터-디테일 톤)

### 7.1 조직도 (`/gw/orgchart`)
- 좌: 부서 트리(접기/펼치기, 상위→하위). 우: 선택 부서 사원 카드(이름·직책·상급자). 상단 검색.

### 7.2 전자결재 (`/gw/approval`)
- **결재함(좌 탭)**: `받은 결재(대기)` · `상신함(내가 올림)` · `완료함` · `참조함` · `임시저장`.
- **목록(중)**: 문서번호·유형·제목·기안자·상태 배지·현재 결재자.
- **상세(우)**: 본문 + **결재선 진행 타임라인**(각 노드 kind·결재자·상태·의견) + 내 차례면 **[승인][반려][보류]** 액션(반려 사유 입력).
- **상신 모달**: 유형 선택 → **결재선 빌더**(§7.3) → 본문/금액/유형필드 → [임시저장][상신].

### 7.3 결재선 빌더(`ApprovalLineBuilder`, 핵심 컴포넌트)
- 상단 토글: **[자동 상신선]**(상급자 체인 프리필) · **[전결규정 적용]**(유형·금액 기반 절단) · **[수동]**.
- 결재선 리스트: 각 노드 = 결재자(피커) + `kind` 선택(결재/합의/참조/전결/대결) + 순서 드래그 + 병렬 묶기. 삭제/추가.
- 실시간 검증: 결재자 0명·본인만 등 방지, 전결권자 존재 확인.

### 7.4 휴가관리 (`/gw/leave`)
- 상단: **잔여 요약**(부여/사용/잔여 도출). 중: 신청 폼(종류·기간→일수 자동, 결재선=자동 상신선 기본). 하: 내 휴가 내역(상태 배지, 결재 진행 링크).

### 7.5 도크 연동
- GroupwarePanel: 전자결재 타일 배지·"결재 대기" 카드를 `useGwSummary` 도출값으로 교체. 프로필=로그인 사용자. 항목 클릭 → `/gw/approval` 딥링크.

## 8. 단계별 로드맵

| Phase | 산출물 | 완료 기준(DoD) |
|---|---|---|
| **0. 도크 포털 실동작화** | GroupwarePanel 목업 제거, 프로필=로그인 사용자, 타일→`/gw/*` 라우팅, `useGwSummary` 골격 | 타일 클릭 라우팅·도크 닫힘, 프로필 실사용자, tsc·build 통과 |
| **1. 조직도 + 조직 데이터** | `users.managerId` 추가·시드, `departments` 도메인/repo/seed, `useOrgTree`, 조직도 화면 | 부서 트리·상급자 체인·부서장 도출, 조직도 렌더, build 통과 |
| **2-a. 결재 엔진(도메인)** | `approvalDoc`·`approvalRule` 스키마, **순수 상태전이 함수**(`applyDecision`·전결 skip·병렬 합의), repo(box/submit/approve/reject/recall/delegate) + seed | 함수 단위 전이 정확(순차·병렬·전결·반려·회수), repo dry-run 로드 |
| **2-b. 결재 UI** | 결재함·상세 타임라인·상신 모달·**결재선 빌더(3방식)**, 도크 대기 배지 실데이터 | 상신→승인/반려/회수→완료 반영, 자동상신선·전결규정·수동 결재선 구성, 배지 도출 정확 |
| **3. 휴가관리** | 휴가 `form`·잔여 도출·`useLeave`, 휴가 화면(신청→결재 상신) | 휴가 신청이 결재 엔진으로 흐름, 잔여 도출, 승인 완료 반영 |
| **4. 영속화** | 1~3 repo Firestore 연결, rules/index/시드러너 통합 | 새로고침·다계정에서 결재·휴가 유지, 대기 배지 정확 |

- 순차 진행. **2-a(엔진)와 2-b(UI) 분리** — 엔진을 순수함수로 먼저 확정하면 UI·DB가 안전하게 얹힘.
- 검증: 각 Phase 후 `/run` 또는 로컬 구동 육안 + **다계정 결재 시연**(A 상신 → B 결재자 로그인 승인 → A 완료 확인, Phase 4).
