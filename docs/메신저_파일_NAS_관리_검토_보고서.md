# 메신저 첨부파일 NAS 관리 — 가능성 검토 보고서

> 작성일: 2026-07-15 · 대상: 사내 메신저 첨부파일의 저장소를 Firebase Storage → 사내 NAS로 전환
> 관련 문서: `DEMO.md`(tus 파일서버), `메신저_개발_계획서.md`, `DB_이관_대비_설계원칙.md`
> 근거 코드: `src/data/chatMessage/chatMessage.repo.ts`, `src/domain/chatMessage/schema.ts`, `storage.rules`, `src/app/shell/QuickDock.tsx`

---

## 0. 결론 (요약)

**가능하다. 다만 "설정 하나로 되는 일"은 아니고, 별도 파일서버 구축 + 인증·다운로드 방식 재설계가 필요한 중규모 작업이다.**

- ✅ **구조적으로 유리하다.** 현재 첨부 저장은 `chatMessageRepo.uploadAttachment()` 한 곳에 캡슐화되어 있고, 메시지는 저장 위치를 모른 채 `{url, name, size, mime}` 메타만 들고 있다. 저장소를 NAS로 바꿔도 **화면·도메인·메시지 스키마는 무변경**, repo 1개 함수 + 다운로드 함수만 교체하면 된다.
- ⚠️ **선결 과제 3가지**가 있다: (1) `DEMO.md`가 설명하는 파일서버가 **아직 이 저장소에 없다**(구축·배포 필요), (2) 앱의 자체 로그인과 파일서버 JWT 인증의 **인증 모델 불일치**, (3) 파일서버의 **1인 소유(owner) 격리 모델**이 메신저의 **방 참여자 공유** 접근과 맞지 않는다.
- 📌 **권고**: NAS 전환은 "대용량·사내보관·비용" 요구가 실재할 때 가치가 있다. 단순 데모/소용량이면 현행 Firebase Storage 유지가 합리적. 전환 시 아래 §6 하이브리드(추상화) 방식으로 단계 진행을 권고한다.

## 1. 현재 구조 — 메신저 첨부는 지금 어디에 저장되나

| 항목 | 현재 구현 | 위치 |
|---|---|---|
| 업로드 | Firebase Storage `chat/{roomId}/{timestamp}-{파일명}` | `chatMessage.repo.ts:55-80` |
| 폴백 | Firebase 미설정 시 **base64 data URL**(메시지 문서에 인라인) | `chatMessage.repo.ts:73-79` |
| 저장 메타 | `{ url, name, size, mime }` 만 메시지 문서에 기록 | `schema.ts:19-29` |
| 용량 제한 | **10MB**(schema·storage.rules·UI 3곳에서 강제) | `schema.ts:16` |
| 접근 제어 | `chat/**` → `allow read: if true` (**URL만 알면 누구나 열림**) | `storage.rules:20-24` |
| 다운로드 | 저장된 `url`로 앵커 클릭, `Content-Disposition` 헤더로 원본 파일명 보존 | `QuickDock.tsx:903-911` |
| 버킷 | `workfit-office-app.firebasestorage.app` | `.env.local` |

**핵심**: 메시지 문서는 파일이 **어디 있는지 모른다.** 그저 `url`을 가리킬 뿐이다. 이 URL이 Firebase든 NAS든 앱 렌더링·다운로드 로직에는 차이가 없다 → **이것이 NAS 전환을 가능케 하는 결정적 지점.**

## 2. NAS 방식이란 — `DEMO.md`가 설명하는 파일서버

`docs/DEMO.md`는 **별도의 tus 기반 파일서버**를 설명한다(현재 저장소에는 코드 미포함, 계획/설계 단계):

- **저장**: 로컬 `./data` (데모) → **NAS 볼륨 마운트** (운영). `USE_XACCEL` 환경변수 하나로 전환.
- **다운로드**: 운영에서 nginx `X-Accel-Redirect`로 NAS 파일을 흘려보냄(Node 이벤트 루프 보호).
- **인증**: JWT + DB `id AND owner_id` 조건으로 **사용자별 파일 격리**.
- **강점**: tus 프로토콜로 **이어올리기(resumable)** — 대용량 파일에 강함. 청크 단위 업로드.
- **운영 요건**: HTTPS(Let's Encrypt), 역방향 프록시, DSM 전용 계정, 쿼터(5GB), 야간 정리, Hyper Backup.

→ NAS 전환 = 이 파일서버를 구축·배포하고, 메신저 업로드를 Firebase Storage 대신 이 서버로 보내는 것.

## 3. 가능성 판단 — 왜 가능한가

정본 데이터 계층 패턴([[DB_이관_대비_설계원칙.md]]) 덕분에 **교체 지점이 좁다.**

```
[변경 없음] QuickDock UI · useSendAttachment · chatMessageSchema · 메시지 문서
                          │  (여전히 {url,name,size,mime} 만 소비)
                          ▼
[여기만 교체] chatMessageRepo.uploadAttachment()  →  NAS 파일서버로 POST
             downloadAttachment()                →  토큰 기반 다운로드로 변경
```

- 업로드: `uploadAttachment(roomId, file)`가 Firebase `uploadBytes` 대신 파일서버에 `fetch(POST)` 하고, 응답의 파일 URL을 `{url, name, size, mime}`로 반환하도록 바꾸면 끝. **호출부(useSendAttachment)는 무변경.**
- 다운로드: 현재는 공개 URL 앵커 클릭. NAS는 보호된 경로라 **토큰/서명 URL** 또는 프록시가 필요 → `downloadAttachment()` 수정.
- 스키마: `attachment: {url,...}` 그대로. URL 대상만 바뀜.

**즉, 아키텍처는 이미 NAS 전환을 염두에 둔 형태다.**

## 4. 핵심 쟁점 — 반드시 풀어야 할 것

| # | 쟁점 | 내용 | 대응 방향 |
|---|---|---|---|
| 1 | **파일서버 부재** | `DEMO.md`의 fileserver가 저장소에 없음 | 서비스 구축·NAS 마운트·배포 (별도 인프라 작업) |
| 2 | **인증 모델 불일치** | 앱은 자체 로그인(인메모리, Firebase Auth·실토큰 없음). 파일서버는 JWT 필요 | 파일서버용 토큰 발급 경로 신설(로그인 세션 → 파일서버 JWT 교환) |
| 3 | **격리 모델 충돌** | 파일서버는 **1인 owner_id** 격리. 메신저 첨부는 **방 참여자 전원 공유** | 소유권을 user가 아닌 `roomId`+`members` 기준으로 재설계 |
| 4 | **다운로드 접근 제어** | 현재 `chat/**`은 공개(URL만 알면 열림). NAS는 보호 경로 | 서명 URL/토큰 다운로드 or 앱 경유 프록시로 방 멤버만 허용 |
| 5 | **네트워크 노출** | 브라우저 → NAS 직접 업로드 시 CORS·HTTPS·역방향 프록시 필요 | nginx 리버스 프록시 + Let's Encrypt(사외 접근 시) |
| 6 | **가용성·백업** | Firebase=관리형 다중화. NAS=단일 장애점 | RAID + Hyper Backup 외부 백업 필수 |
| 7 | **폴백 정합** | Firebase 미설정 폴백은 base64 인라인. NAS 경로의 폴백 정의 필요 | 파일서버 미가용 시 동작(에러 or 인라인) 명시 |

가장 무거운 것은 **#2·#3·#4(인증·격리·접근제어)** — NAS의 "1인 소유 파일함" 모델과 메신저의 "방 공유 첨부" 모델이 근본적으로 다르다. 파일서버 인가 로직을 방 멤버십 기준으로 다시 짜야 한다.

## 5. 방안 비교

| 기준 | A. 현행 Firebase Storage | B. NAS 파일서버 전환 | C. 하이브리드(추상화 후 선택) |
|---|---|---|---|
| 구축 공수 | 0 (완료) | 높음(서버+인프라+인증) | 중(추상화 계층 먼저) |
| 대용량/이어올리기 | 10MB 제한, 단발 | tus 이어올리기, 대용량 ○ | 저장소 선택에 따름 |
| 사내 보관/데이터 주권 | 클라우드 | **사내 NAS(온프렘)** | 선택 가능 |
| 비용 | GB당 클라우드 과금 | NAS 하드웨어(고정) | — |
| 가용성/백업 | 관리형(높음) | 자체 RAID+백업 필요 | — |
| 접근 제어 | 현재 공개(취약) | 토큰 기반(강화 가능) | 강화 가능 |
| 운영 부담 | 낮음 | 높음(HTTPS·정리·백업·계정) | 중 |

## 6. 권고 — 단계별 전환 로드맵

**"저장소 추상화 → 파일서버 구축 → 스위치" 순서**로, 되돌릴 수 있게 진행.

- **Phase 0 · 저장소 추상화(코드 내, 저비용)**
  `uploadAttachment`/`downloadAttachment`를 저장소 드라이버 인터페이스로 분리(`firebase` | `nas`), 환경변수로 선택. 이 단계만으로 화면·스키마 영향 0. 이후 어떤 백엔드든 꽂을 수 있는 상태 확보.

- **Phase 1 · 파일서버 구축(인프라)**
  `DEMO.md` 기반 tus 파일서버를 NAS 볼륨에 배포. HTTPS·역방향 프록시·전용 계정. **먼저 단순 파일함 데모로 검증**(DEMO.md 체크리스트).

- **Phase 2 · 인증·격리 재설계(핵심 난제)**
  - 로그인 세션 → 파일서버 JWT 발급 경로.
  - 소유권을 `owner_id` → `roomId`+`members` 기반 인가로 변경.
  - 다운로드를 방 멤버 토큰 검증(공개 URL 폐기).

- **Phase 3 · 메신저 연결·스위치**
  `nas` 드라이버를 메신저 첨부에 연결, 파일럿 방에서 시연 → 문제없으면 기본값 전환. 롤백은 환경변수 원복.

- **Phase 4 · 운영화**
  쿼터·야간 정리(tus 사이드카 포함)·Hyper Backup·용량 모니터링.

## 8. 종합 판단

- **기술적 가능성: ○** — 저장소가 repo 한 지점에 캡슐화되어 있어 교체 지점이 좁다. 아키텍처가 이미 이 전환을 전제한다.
- **실제 난이도: 중~상** — 코드 변경은 작지만, **파일서버 구축 + 인증/격리 재설계 + 운영(HTTPS·백업·정리)**이 실제 일감이다. "설정 플래그 하나"가 아니다.
- **추천**: 대용량·사내 데이터 보관·클라우드 비용 회피가 실제 요구라면 전환 가치가 크다(tus 이어올리기·온프렘 통제는 명확한 이득). 그렇지 않다면 현행 유지가 합리적. 진행한다면 **Phase 0 추상화부터** 착수해 위험을 뒤로 미루는 것을 권고한다.
