# 경영자 대시보드(Executive Cockpit) 개발 계획서

> 작성일: 2026-07-03 · 대상: 경영진/공장장이 보는 최상위 현황 화면
> 관련 문서: `데이터_계층_구현현황.md`, `데이터_모델_설계서.md`, 메모리 `data-layer-pattern`

---

## 1. 배경 · 문제 정의

경영자가 볼 화면을 "새로 그리는" 문제가 아니다. **이미 화면은 있으나 가짜 숫자**라는 것이 핵심 문제다.

- 리포트 모듈에 **경영 대시보드 그룹(`G_REP_EXEC`) 3종이 이미 존재**한다.
  - `/report/kpi` — 종합 KPI 대시보드 (`ReportKpiScreen`)
  - `/report/period` — 기간 비교 트렌드 (`ReportPeriodScreen`)
  - `/report/scorecard` — 부서/라인 스코어카드 (`ReportScorecardScreen`)
  - 정본 와이어프레임: `mes layout/mes/report-exec.jsx`
- 그러나 세 화면 모두 **컴포넌트 상단에 하드코딩된 mock 배열**을 렌더한다. 예: `ReportKpiScreen`의 `tiles`/`scores`는 상수. 이미 구축된 **118개 실데이터 컬렉션과 연결되어 있지 않다.**
- 운영용 `통합 모니터링`(`/ops/dashboard`, `DashboardScreen`)도 mock(`mock.ts`) 상태.

**따라서 이 프로젝트의 진짜 과제 = 집계 계층(reporting/aggregation)을 신설해 실데이터로 KPI를 계산하고, 경영자 전용 진입점으로 승격.**

### 운영 화면 vs 경영 화면 (구분)
| 구분 | 통합 모니터링(`/ops/dashboard`) | 경영자 대시보드(본 계획) |
|---|---|---|
| 사용자 | 현장/관리자 | 경영진/공장장 |
| 시간축 | 실시간(초·분) | 일·주·월·전월비 트렌드 |
| 내용 | 라인/설비 상태, 알람 | 재무·생산·품질·납기 요약 KPI, 부문 평가 |
| 조작 | 드릴다운·조치 | 요약 조회 + 이상치 드릴다운 |

---

## 2. 목표

1. 경영자가 **로그인 직후 한 화면에서 전사 상태를 파악**한다(생산·품질·설비·자재·영업·원가).
2. 표시되는 모든 수치는 **실제 repo 데이터에서 집계**된다(mock 제거).
3. 이상 지표는 **색/배지로 즉시 식별**되고, 클릭 시 원천 화면으로 **드릴다운**된다.
4. 데이터 계층 정본 패턴([[data-layer-pattern]])을 지켜 **DB 교체·실운영 전환에 무영향**이도록 집계를 격리한다.

## 3. 권장 접근안 (Option C · 하이브리드)

세 가지 안을 검토했다.

| 안 | 내용 | 장점 | 단점 |
|---|---|---|---|
| A | 기존 `/report/kpi` 3종을 실데이터로만 연결 | 최소 공수 | 리포트 하위에 묻혀 경영자 진입성 낮음 |
| B | 완전 신규 경영 화면 신설 | 자유도 | 기존 와이어프레임/컴포넌트 낭비, 재작업 |
| **C(권장)** | **집계 계층 신설 + 기존 exec 컴포넌트 재사용 + 최상위 "경영 현황" 진입점 추가** | 재사용·정본충실·진입성 모두 확보 | 집계 계층 설계 필요(어차피 필수) |

### 권장 구성
1. **집계 계층**(신규, 핵심): 여러 repo를 가로질러 KPI를 계산하는 셀렉터/서비스. → §5
2. **경영 현황 랜딩**(신규 화면): 최상위 메뉴 또는 로그인 후 기본 진입 화면. 기존 exec 카드/차트 컴포넌트(`_report.tsx`의 `RHead/RParam/ProgBar`, `Sparkline/LineChart/Donut/RankBars`)를 재사용해 요약 + 알림 + 드릴다운.
3. **기존 exec 3종 실데이터화**: 같은 집계 계층을 소비하도록 리팩터(mock 제거).

## 4. 화면 정의 (경영 현황 랜딩)

와이어프레임 `report-exec.jsx`(KPI 타일+부문평가+트렌드) + `dash-a.jsx`(상단 요약 스트립) 디자인 언어를 따른다.

```
┌ 경영 현황 (전사 · 2026-06 월간 · [기간/공장 필터]) ───────────────┐
│  [KPI 타일 6~8] 생산달성률·수율(FPY)·평균OEE·납기준수율          │
│                 재고회전율·클레임·월매출·원가율                   │
│                 (각 타일: 값 + 전월비 ▲▼ + 스파크라인)           │
├───────────────────────────────┬───────────────────────────────┤
│ 생산성 vs 품질 (월별 라인)     │ 부문별 종합 평가 (ProgBar)     │
│                                │ 생산/품질/설비/자재/원가 점수  │
├───────────────────────────────┼───────────────────────────────┤
│ 경영 알림(Exec Alerts)         │ 이번 달 주요 지표 전월 대비    │
│ · 납기 임박/지연 N건           │ (기간 비교 표)                 │
│ · 안전재고 미달 N건            │                                │
│ · 미해제 품질 알람 N건         │                                │
│ · 여신 초과 거래처 N건         │ → 각 항목 클릭 = 원천 화면 이동│
└───────────────────────────────┴───────────────────────────────┘
```

- **KPI 타일**: 클릭 시 해당 상세 리포트로 드릴다운.
- **경영 알림**: 각 항목은 이미 있는 파생상태(안전재고 미달 `safetyStock`, 품질 알람 `spcAlarms`/`liveAlarms`, 여신 `creditLimits`, 납기 `salesOrders.deliveryStatus`)를 카운트.

## 5. 데이터 · 집계 계층 설계 (핵심)

정본 패턴을 지켜 **화면이 repo를 직접 집계하지 않는다.** 집계는 전용 계층에 격리한다.

### 5-1. 배치
```
src/features/report/useExecKpis.ts   — TanStack Query 훅(경영 랜딩·exec 3종이 소비)
src/services/reporting.service.ts    — 여러 repo를 가로지르는 집계(순수 계산은 domain로 분리 가능)
src/domain/report/kpi.ts             — 순수 집계 함수(입력 배열→KPI 숫자). 테스트 용이·DB 무관
```
- `reporting.service`가 필요한 repo(`workOrderRepo`, `inspectionRepo`, `oeeEquipmentRepo`, `salesRevenueRepo`, `stockRepo`, `spcAlarmRepo`, `safetyStockRepo`, `creditLimitRepo` 등) `list()`를 병렬 로드 → `domain/report/kpi.ts`의 순수 함수로 집계.
- **cross-entity 서비스 계층 규칙과 동일**: repo를 조합하므로 `services/`에 둔다([[data-layer-pattern]] 패턴 6). 운영 전환 시 이 함수 내부만 Firestore 집계쿼리/Cloud Function/사전집계 컬렉션으로 교체하면 화면 무변경.
- 캐시: exec KPI는 무거우므로 `staleTime` 상향(예 5분), 월간 집계는 `period` 키로 분리.

### 5-2. KPI 카탈로그 (지표 → 원천 → 산식 초안)
| KPI | 원천 컬렉션 | 산식(초안) |
|---|---|---|
| 생산 달성률 | `productionResults` / `prodPlans` | Σ실적수량 ÷ Σ계획수량 |
| 종합 수율(FPY) | `inspections`/`prodDefects` | 양품 ÷ 투입 (초·중·종/검사 합격 기준) |
| 평균 OEE | `oeeEquipments` | 설비 OEE 평균(또는 가동시간 가중) |
| 납기 준수율 | `salesOrders`(파생 `deliveryStatus`)/`shipments` | 납기내 출하 ÷ 전체 납기 |
| 재고 회전율 | `stockMovements`/`stocks` | 출고금액 ÷ 평균재고금액 |
| 클레임 건수 | `voc` | 기간 내 접수 건수 |
| 월 매출 | `salesRevenues` | Σ당월 매출 |
| 원가율 | `cost`(리포트)·`productionResults` | 제조원가 ÷ 매출 |
| 부문 점수 | 위 지표 가중합 | 부문별 정규화 점수(0~100) |

> ⚠ 산식은 **초안**이다. 확정 전 경영진 정의(수율 기준 시점, OEE 가중, 원가 산정)를 §9에서 확인 필요.

### 5-3. 데이터 방침 (확정: 데모 = 샘플 데이터 허용)
**결정(2026-07-03): 현재는 데모이므로 대시보드 수치를 샘플 데이터로 작성해도 된다.** 따라서 무거운 실데이터 집계 계층은 **필수가 아니라 후속(P5)** 으로 미룬다.

- **1차 구현**: 경영자 화면에 맞는 **구조화된 샘플 데이터 모듈**(`src/modules/exec/mock.ts` 등)로 화면을 완성한다. 기존 exec 3종이 쓰던 방식과 동일하되, 회사 전체를 대표하도록 카테고리별로 정리한다.
- **집계 계층은 후속**: §5-1의 `domain/report/kpi` + `services/reporting`는 실운영 데이터로 전환할 때 도입한다. **단, 화면은 처음부터 "데이터 소스 함수" 하나(예: `getExecDashboardData()`)만 호출**하도록 만들어, 나중에 그 함수 내부만 샘플→실집계로 바꾸면 화면 무변경이 되게 격리한다.
- 샘플 값은 seed(companyInfo·부서·라인 등 실제 존재 값)와 어긋나지 않게 톤을 맞춘다.

## 6. 정보구조 · 접근/권한 (확정: 로그인 랜딩)

**결정(2026-07-03): 로그인 직후 진입하는 랜딩 화면 자체를 경영자 대시보드로 만든다.** 회사 전체 상황(생산·품질·설비·자재·영업·원가)을 경영자 관점 한 화면에 요약해 첫 화면으로 노출한다.

- **진입점**: 최상위 경로 `/exec`(예: 메뉴 `M_EXEC` order=5, 이름 `경영 현황`). 로그인 성공 → 기본 리다이렉트 대상.
- **랜딩 라우팅**: `AuthGate` 통과 후 셸의 초기 화면을 `/exec`로. 현재 로그인 후 이동 로직(초기 활성 화면/라우트)이 어디서 정해지는지 확인해 그 지점만 교체.
- **역할 범위(확정)**: **전 사용자 공통 랜딩.** 누가 로그인하든 첫 화면이 경영자 대시보드다(데모 성격). 역할별 분기는 후속으로 열어둔다.
- 기존 `G_REP_EXEC` 3종(KPI/기간비교/스코어카드)은 **드릴다운 상세 목적지**로 유지(랜딩 요약 → 상세).

## 7. 구현 단계 (마일스톤)

**샘플 데이터 우선 순서**(데모 방침 반영). 화면부터 완성하고 실집계는 마지막.

| 단계 | 산출물 | 검증 |
|---|---|---|
| **P1 데이터 소스 + 화면** | `src/modules/exec/`(`ExecDashboardScreen.tsx` + `mock.ts`의 `getExecDashboardData()` 단일 소스). 기존 exec 카드/차트 컴포넌트 재사용. KPI 타일 + 부문 평가 + 경영 알림 + 트렌드 | 스크린샷(로그인→랜딩) |
| **P2 랜딩 라우팅** | `/exec` 라우트 등록 + 최상위 메뉴 `M_EXEC` + **로그인 후 기본 진입을 `/exec`로**(전 사용자) | 로그인 시 첫 화면 확인 |
| **P3 드릴다운** | KPI 타일·경영 알림 클릭 → 원천 화면 이동 연결 | 클릭 이동 확인 |
| **P4(후속) exec 3종 정리** | 기존 `/report/kpi·period·scorecard`를 같은 소스로 통일 | 값 일치 |
| **P5(후속) 실집계 전환** | `getExecDashboardData()` 내부만 샘플→`services/reporting` 실집계로 교체(화면 무변경), 기간 필터 실동작 | seed/실데이터 기준 값 |

각 단계 종료 시 `npm run build`(tsc 0에러) + 커밋/푸시 → Vercel 자동배포(현 파이프라인).

## 8. 기술 결정 · 리스크

- **집계 격리(필수)**: 화면·domain은 firebase 무관. repo만 조합하는 서비스에 집계 집중 → 운영 전환 시 화면 무변경.
- **성능**: 다수 컬렉션 `list()` 전체 로드 집계는 데모 규모(685건)엔 무해. 실운영 확장 시 **사전집계(daily rollup) 컬렉션 + 증분 갱신**으로 이행(P5). 이 경계도 서비스 내부라 화면 영향 없음.
- **와이어프레임 충실도**([[wireframe-source-of-truth]]): 경영 랜딩은 전용 와이어프레임이 없다 → `report-exec.jsx`+`dash-a.jsx` 디자인 언어를 조합해 신규 구성(운영현황 라인가동 화면 선례와 동일 방식). 진행 전 방식 확인 대상.
- **산식 신뢰성**: 잘못된 산식은 경영 오판을 부른다. P0에서 지표 정의 문서화 후 착수.

## 10. 요약

- 경영 화면은 **이미 UI가 있으나 mock**이다 → 진짜 일은 **집계 계층 신설 + 실데이터 연결 + 경영자 진입점 승격**.
- 권장: 집계 계층(`domain/report/kpi` + `services/reporting` + `features/report`) 위에, 기존 exec 컴포넌트를 재사용한 **최상위 `경영 현황` 랜딩**을 얹고, 기존 리포트 exec 3종을 상세로 실데이터화.
- 데이터 계층 정본 패턴을 지켜 **실운영 전환에 화면 무변경**을 보장.
- 착수 전 §9의 5개 결정(특히 진입점·KPI 산식)을 확정한다.
