# 사내 메신저(Messenger) 개발 계획서

> 작성일: 2026-07-06 · 대상: 퀵 도크 우측 슬라이드 패널의 "메신저" 기능
> 관련 문서: `데이터_계층_구현현황.md`, `데이터_모델_설계서.md`, `DB_이관_대비_설계원칙.md`, 메모리 `data-layer-pattern`
> 정본 와이어프레임: `mes layout/mes/quick-dock.jsx` (목록 화면만 존재 — 대화창은 신규 설계)

---

## 1. 배경 · 현황

퀵 도크(`src/app/shell/QuickDock.tsx`)에는 그룹웨어·위디(챗봇)·**메신저** 3종 도구가 있다. 이 중 메신저를 실제 사용 가능한 기능으로 개발한다.

**현재 상태 = 껍데기 목록.**
- `MessengerPanel`(`QuickDock.tsx:231-266`)은 **하드코딩된 채팅방 6개**(`chats` 배열)를 렌더한다.
- 채팅방 버튼에 **`onClick` 핸들러가 없다** → 클릭해도 아무 일도 일어나지 않는다.
- **대화창(thread)·메시지 전송·미읽음 갱신·실시간 수신이 전혀 없다.**
- 상단 검색바도 정적 텍스트(비기능).

**정본 와이어프레임 확인 결과**(`quick-dock.jsx`): 메신저는 **채팅방 목록 화면만** 정의돼 있고, 방을 열었을 때의 **대화창 디자인은 와이어프레임에 없다.** → 목록은 1:1 이관(완료 상태), **대화창·입력창은 신규 설계 영역**이며, 같은 파일의 `ChatbotPanel`(위디) 말풍선/입력창 스타일을 디자인 언어 기준으로 재사용한다.

**아키텍처 현황**: 앱 전체가 `domain → data(repo) → features(React Query) → modules` 정본 패턴([[data-layer-pattern]])으로 118개 컬렉션을 운용한다. 단 **실시간 구독(`onSnapshot`)은 코드베이스에 0건** — 모든 조회는 1회성 `getDocs` + React Query 캐시다. 메신저는 본질적으로 실시간이므로 **이 프로젝트의 핵심 기술 결정 = 실시간 수신을 어떻게 처리할 것인가**이다(§4).

---

## 2. 목표

1. 도크에서 채팅방을 **클릭하면 대화창이 열리고**, 지난 대화가 시간순으로 보인다.
2. 메시지를 **입력·전송하면 대화에 추가**되고(내 말풍선), 상대 메시지가 **수신·표시**된다.
3. 채팅방 목록의 **마지막 메시지·시간·미읽음 배지가 실제 데이터로 갱신**된다.
4. 데이터 계층 정본 패턴을 지켜 **repo만이 Firestore를 알고**, DB 교체·실운영 전환에 화면·도메인이 무영향([[DB_이관_대비_설계원칙.md]]).
5. Firebase 미설정 시에도 **in-memory seed로 graceful degrade**(앱이 데모로 계속 동작)한다.

### 비목표(이번 범위 제외)
- 파일·이미지 첨부, 이모지 리액션, 음성/영상 통화, 멘션 알림, 메시지 검색 인덱싱.
- 종단 암호화·실제 보안 인증(데모 한정, users 평문 비밀번호와 동일한 데모 전제).

> **확정 결정(2026-07-06)**: 실시간은 **Phase 2 폴링**까지 · 상대 응답은 **다계정 로그인 시연** · **신규 방 생성 포함**. → 아래 범위·로드맵에 반영.

## 3. 범위 정의 (데모 현실성 기준)

| 항목 | 이번 범위 | 비고 |
|---|---|---|
| 채팅방 목록 | ✅ 실데이터화 | 마지막 메시지·시간·미읽음 도출 |
| 대화창 열기/닫기 | ✅ | 목록 ↔ 대화창 패널 내 전환 |
| 메시지 전송 | ✅ | 낙관적 추가(optimistic) + 저장 |
| 메시지 수신(실시간) | ✅ | 방식은 §4에서 결정 |
| 미읽음 카운트 | ✅ 도출 | 방별 마지막읽음 vs 마지막메시지 |
| 시스템 메시지(공지방) | ✅ | type='system' 렌더 분기 |
| **신규 방 생성(1:1/그룹)** | ✅ 포함(확정) | 조직도/사용자 목록에서 상대 다중 선택 → 방 생성. `members` N명 = 그룹방 |
| **기존 방에 그룹초대** | ✅ 포함(확정) | 대화창에서 멤버 추가 → `members` append + 시스템 메시지 |
| **다계정 시연** | ✅ 전제 | 두 계정 로그인해 실제 송수신 시연 → Phase 2 영속·폴링 필수 |
| 첨부/통화/리액션 | ⛔ | 비목표 |

## 4. 핵심 결정 — 실시간 수신 방식

세 가지 안을 검토한다. **repo가 유일한 Firebase 계층**이라는 원칙은 어느 안에서도 유지된다(§6).

| 안 | 방식 | 장점 | 단점 |
|---|---|---|---|
| A | **React Query `refetchInterval` 폴링**(3~5초) | 기존 패턴 그대로, 신규 인프라 0, repo는 `getDocs` 유지 | 진짜 실시간 아님(수 초 지연), 유휴 시 불필요 요청 |
| B | **Firestore `onSnapshot` 구독**(신규 계층) | 진정한 실시간, Firestore 네이티브, 효율적 | 코드베이스 첫 구독 계층 — repo에 `subscribe()`·구독 해제 수명주기 신설 필요 |
| C | 로컬 전용 인터랙티브 mock(비영속) | 최소 공수 | 새로고침 시 초기화, 다중 사용자·다중 탭 불가 |

### 확정: **Option A 폴링 (Phase 2까지)**

- **Phase 1 (UX 우선)**: seed/in-memory 위에서 **대화창·입력·낙관적 전송**을 완성해 인터랙션을 먼저 증명. 목록·대화는 React Query 캐시 무효화로 갱신(폴링 없이 로컬 동작).
- **Phase 2 (영속 + 준실시간, 최종 목표)**: repo를 Firestore로 연결하고 **폴링(`refetchInterval` 3~5초)**으로 세션·탭·계정 간 반영. **다계정 로그인 시연**(계정 A가 보낸 메시지를 계정 B 브라우저가 폴링으로 수신)이 이 위에서 동작한다. 최소 리스크로 "진짜 저장·수신" 달성.
- **Phase 3 (`onSnapshot`)**: 이번 범위 제외. 폴링이 데모에 충분하며, 추후 필요 시 repo에 `subscribe`만 추가해 화면·도메인 무변경으로 승격 가능(경로만 열어둠).

이 접근은 앱의 graceful-degrade 철학과 일치하고, 다계정 시연에 필요한 영속·수신을 폴링으로 충족한다.

## 5. 데이터 모델

정본 데이터 모델 카탈로그([[데이터_모델_설계서.md]])에 **채팅 도메인 2종을 신설**한다.

### 5.1 `chatRooms` — 채팅방 (마스터)
`src/domain/chatRoom/schema.ts` (zod SSOT)

| 필드 | 타입 | 설명 |
|---|---|---|
| `id` | string (PK) | 방 ID (`RM-xxxx`) |
| `name` | string | 방 이름(그룹) / 상대 이름(1:1) |
| `type` | enum `group\|direct\|notice` | 그룹/1:1/공지방 |
| `members` | string[] | 참여자 `users.id` 배열 (FK) |
| `color` | string | 아바타 액센트(목록 렌더) |
| `lastMessage` | `{ text, at, senderId }` \| null | **비정규화** — 목록 표시·정렬용 |
| `createdAt` | string(ISO) | 생성 시각 |

- 미읽음·마지막메시지는 목록에서 **매번 메시지를 훑지 않도록** `lastMessage`를 방 문서에 비정규화(운영 전환 시 트리거/함수가 유지, 로직 불변 — stock 원장→스냅샷 패턴과 동일 사상).
- **그룹방 = `type:'group'` + `members.length ≥ 3`**(1:1은 `direct`, 정확히 2명). 방 생성 시 상대를 여러 명 고르면 그룹, 한 명이면 1:1로 자동 판정. **그룹초대**는 기존 방 `members`에 id를 append(방 문서 1건 갱신) → 참여자 수·아바로 즉시 반영. `notice`(공지방)는 초대·전송 UI 없이 읽기 전용 시스템 방.

### 5.2 `chatMessages` — 메시지 (로그, 플랫 top-level 컬렉션)
`src/domain/chatMessage/schema.ts` — 저장 위치 **플랫 top-level 컬렉션 `chatMessages`**(`roomId` 필드로 방 구분).
> **구현 결정(Phase 0)**: 계획 초안은 서브컬렉션(`chatRooms/{id}/messages`)이었으나, **코드베이스 118개 컬렉션에 서브컬렉션 선례가 0건**이고 시드러너가 `{coll, docs, id}` 균일 매핑이라 **플랫 컬렉션으로 확정**. 방별 조회는 `roomId` 필터(vendor 전건 로드+필터 패턴과 동일). 추후 규모 필요 시 `where('roomId','==')` + 인덱스로 승격.

| 필드 | 타입 | 설명 |
|---|---|---|
| `id` | string (PK) | 메시지 ID |
| `roomId` | string | 소속 방 (FK) |
| `senderId` | string | 보낸 사람 `users.id` |
| `senderName` | string | 비정규화(렌더용) |
| `text` | string | 본문 |
| `type` | enum `text\|system` | 일반/시스템(공지·입장 안내) |
| `at` | string(ISO) / serverTimestamp | 전송 시각(정렬 키) |
| `readBy` | string[] | 읽은 사용자 id (미읽음 도출) |

- **미읽음 카운트 = 도출값**(저장 안 함): 방의 `lastMessage.at`과 사용자별 마지막 읽음 시각 비교, 또는 `readBy`에 현재 uid 미포함 메시지 수. → 데이터-상태 불일치 불가능([[data-layer-pattern]] 파생상태 원칙).
- 시각 정렬: 앱 런타임 `Date.now()`/serverTimestamp 사용(런타임 코드에서 허용 — 채번 예약 규칙은 counters 시퀀스에만 적용).

### 5.3 현재 사용자·참여자 소스
- 보낸 사람·참여자 이름/부서는 **기존 `users` 도메인**(`src/domain/user`, `src/data/user/user.repo.ts`)에서 조회.
- "나"(발신자)는 `AuthProvider`의 로그인 세션(`mes.auth.uid`)에서 획득.

## 6. 계층 설계 (정본 패턴 준수)

```
src/domain/chatRoom/schema.ts        — 방 zod 스키마(SSOT)
src/domain/chatMessage/schema.ts     — 메시지 zod 스키마(SSOT)
src/data/chatRoom/chatRoom.repo.ts   — 방 list/get + create(1:1/그룹) + addMembers(초대) + lastMessage 갱신. Firebase는 여기서만
src/data/chatMessage/chatMessage.repo.ts
        — 메시지 list(방별·시간순)/append/markRead
src/data/seeds/chatRoom.seed.ts      — 방 6건(현 mock 이관)
src/data/seeds/chatMessage.seed.ts   — 방별 대화 seed(시연용 대화 흐름)
src/features/chat/useChatRooms.ts    — 방 목록(useQuery, 폴링) + 미읽음 도출 + useCreateRoom/useInviteMembers(useMutation)
src/features/chat/useChatThread.ts   — 방 메시지(useQuery, 폴링) + useSendMessage(useMutation, 낙관적)
src/app/shell/QuickDock.tsx          — MessengerPanel 인라인 mock 제거, 훅 소비 + 대화창 뷰 + 방생성/초대 모달
```

- **repo 가드**: `isFirebaseConfigured && db ? Firestore : in-memory seed`. `db`는 `let` export이므로 클로저 진입 전 `const fdb = db;` 캡처(기존 TS 함정 회피, [[data-layer-pattern]] 기록).
- **화면·domain·features는 firebase를 모른다.** 저장소 교체 시 repo만 재작성(추후 onSnapshot 승격도 repo·훅에 국한).
- **낙관적 전송**: `useSendMessage`가 캐시에 즉시 append → 실패 시 롤백 → 성공 시 방 `lastMessage` 무효화(목록 동기 갱신). React Query 표준 패턴.
- **방 생성/초대**: `useCreateRoom`(선택 멤버로 방 doc 생성 + 첫 시스템 메시지) / `useInviteMembers`(`addMembers` + "○○님을 초대했습니다" 시스템 메시지 append). 성공 시 방 목록·해당 방 스레드 캐시 무효화.
- **인프라 통합 파일**(메인이 직접, 서브에이전트 분담 시 충돌 방지): `firestore.rules`에 `chatRooms`/`messages` 규칙, `firestore.indexes.json`에 방별 `at` 정렬 인덱스, `scripts/seed-firestore.ts` 시드러너에 2컬렉션 매핑 추가.

## 7. UI 설계

패널은 **목록 뷰 ↔ 대화 뷰** 2-state로 전환(같은 `<aside>` 내부 스위치, 도크 폭 384px 유지).

### 7.1 목록 뷰 (기존 개선)
- 현 `MessengerPanel` 레이아웃 유지, `chats`를 `useChatRooms()` 결과로 교체.
- 각 방 버튼에 **`onClick` → 대화창 열기** 부여.
- 미읽음 배지·마지막메시지·시간을 도출값으로 렌더.
- 검색바: 이름/방 필터(클라이언트 측 문자열 필터, 경량).

### 7.2 대화 뷰 (신규)
- 상단: ← 뒤로(목록 복귀) + 방 이름/참여자 수 + 색 아바타 + **그룹방일 때 "＋ 초대" 버튼**(`type:'direct'`는 숨김, `notice`는 읽기전용).
- 본문: 시간순 말풍선. **내 메시지 우측(파랑), 상대 좌측(아바타+이름)** — 그룹방은 상대 말풍선에 보낸사람 이름 표기. `type='system'`은 가운데 회색 캡슐(입장/초대 안내). → `ChatbotPanel` 말풍선 스타일(`QuickDock.tsx:207-217`) 재사용.
- 최신 메시지로 자동 스크롤, 신규 수신 시 하단 유지.
- 하단 입력창: 텍스트 입력 + 전송 버튼(엔터 전송). `ChatbotPanel` 입력창 스타일 재사용하되 **실제 상태 바인딩**.

### 7.3 방 생성 · 그룹초대 모달 (신규)
- 목록 뷰 상단 **"＋ 새 채팅"** → 사용자/조직 목록 모달. **다중 선택**(체크) → 확인 시 `useCreateRoom`(2명=1:1, 3명↑=그룹).
- 대화 뷰 헤더 **"＋ 초대"** → 같은 선택 모달(이미 `members`인 사람은 비활성) → `useInviteMembers`로 방에 추가.
- 사용자 소스는 `users` 도메인, 부서별 그룹핑 표시. 선택 UI는 기존 `Modal`/`Field` 공통 컴포넌트 재사용.

### 7.4 도크 통합
- 도크 탭의 메신저 아이콘에 **총 미읽음 배지** 노출(모든 방 미읽음 합).
- 위디(`bot`)·그룹웨어(`gw`)는 무변경.

## 8. 단계별 로드맵

| Phase | 산출물 | 완료 기준(DoD) |
|---|---|---|
