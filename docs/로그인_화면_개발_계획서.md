# 로그인 화면 개발 계획서

작성일: 2026-07-02 · 대상 앱: `DemoMes_v0` (스마트 MES 생산 플랫폼)

## 1. 목표 및 결정 사항

| 항목 | 결정 | 비고 |
|---|---|---|
| 인증 방식 | **자체 로그인** — `users` 컬렉션 대조 (UI 게이트) | Firebase Auth 미사용. 사용자 결정. |
| 로그인 식별자 | **사번(empNo) 또는 이메일** + 비밀번호 | 둘 다 허용, 하나로 매칭 |
| 화면 디자인 | **회사로고 반영 신규** — `companyInfo.logoUrl` 상단 표시 | 전용 와이어프레임 없음 → 신규 구성 |
| 세션 지속 | localStorage 기반 세션(브라우저 유지) | Firebase Auth 세션 대체 |
| 활성화 토글 | 기존 `VITE_AUTH_ENABLED` 플래그 재활용 | 준비 완료 시 `"true"` |

### 보안 한계 (반드시 인지)
- 자체 로그인은 **UI 게이트일 뿐 진짜 보안이 아니다.** 클라이언트가 비밀번호를 대조하므로 Firestore 보안룰의 `request.auth`를 채우지 못한다.
- 현재 보안룰은 `default-deny + 17컬렉션 개방`(비인증 허용) 상태이므로 이 방식과 정합적이다.
- **진짜 보안(request.auth 연동)은 후속 과제**: Cloud Function으로 custom token 발급 → 향후 하이브리드 전환 시 화면 재사용 가능하도록 구조를 잡는다.

## 2. 현재 상태 (as-is)

- `src/app/auth/` 에 **Firebase Auth 기반** 게이트가 이미 존재 → **교체 대상**:
  - `AuthProvider.tsx` — `signInWithEmailAndPassword` / `onAuthStateChanged` 사용
  - `LoginScreen.tsx` — 이메일/비번 카드형 UI (디자인 참고용으로 재활용)
  - `AuthGate.tsx` — `VITE_AUTH_ENABLED` + `demoMode` 게이트 (**로직 대부분 재사용**)
- `main.tsx` 감싸는 순서: `AuthProvider > AuthGate > QueryProvider > App`
  - ⚠ **구조 문제**: `AuthProvider`/`AuthGate`가 `QueryProvider` **바깥**이라 로그인 화면에서 TanStack Query 훅을 못 쓴다. → 로그인은 `userRepo`/`companyInfoRepo`를 **직접 호출**(권장) 하거나, Provider 순서를 재배치한다.
- `src/domain/user/schema.ts` — **password 필드 없음** → 추가 필요. `status`(사용/잠금/미사용), `lastLogin` 필드는 이미 존재.
- `companyInfo` 데이터 계층(domain/data/features) 존재 — `logoUrl`, `name`, `sysName` 사용 가능.
- `UserMenu.tsx` 로그아웃은 `useAuth().signOutUser` 호출 → 새 컨텍스트에 동일 API 유지 필요(무변경으로 호환).

## 3. 작업 범위 (to-be) — 데이터 계층 패턴 준수

정본 계층 패턴(`domain → data → features → screen`)을 따른다. 로그인은 **인증 로직이라 features 대신 `data/auth` + `app/auth` 컨텍스트**로 배치.

### 3.1 도메인 — `users` 스키마에 password 추가
- `src/domain/user/schema.ts`
  - `userSchema` 에 `password: z.string().default('')` 추가 (평문 저장은 데모 한정, 한계 주석 명기).
  - `userFormSchema`(사용자관리 모달)에 비밀번호 입력/변경 필드 반영 여부 결정 — 최소한 신규 생성 시 초기 비번 설정.
  - 시드(`data/seeds/user.seed.ts`)에 기존 사용자별 초기 비밀번호 부여.

### 3.2 데이터 — 인증 조회
- `src/data/auth/auth.repo.ts` (신규)
  - `authenticate(loginId, password)`: `userRepo.list()`에서 `empNo` 또는 `email === loginId` && `password` 일치 && `status === '사용'` 검증.
    - 실패 케이스 구분: 계정없음 / 비번불일치 / `잠금`·`미사용` 상태.
  - `touchLastLogin(userId)`: 로그인 성공 시 `userRepo.update`로 `lastLogin` 갱신(옵션).
  - Firebase 미설정 시 seed 폴백으로 graceful degrade(기존 repo 가드 그대로 활용).

### 3.3 앱 — 세션 컨텍스트 재작성
- `src/app/auth/AuthProvider.tsx` (**재작성**)
  - Firebase Auth import 제거. 자체 세션 상태(`currentUser: User | null`) 관리.
  - `signIn(loginId, password)` → `authRepo.authenticate` 호출, 성공 시 세션 저장.
  - `signOutUser()` → 세션/스토리지 클리어. **기존 API 시그니처 유지**(UserMenu 무변경).
  - 세션 지속: `localStorage`에 최소 식별자 저장, 초기 로드 시 복원(loading 상태 처리).
  - `demoMode` 개념 유지(Firebase 미설정 시 통과) 또는 자체 로그인은 seed로도 동작하므로 정책 재정의.
- `src/app/auth/AuthGate.tsx` (**소폭 수정**)
  - 기존 `VITE_AUTH_ENABLED` + `loading` + `!user → LoginScreen` 로직 재사용.

### 3.4 화면 — 로그인 UI 신규 (회사로고 반영)
- `src/app/auth/LoginScreen.tsx` (**재작성**)
  - 상단: `companyInfo.logoUrl` 있으면 로고 이미지, 없으면 기존 "M" 폴백 마크 + `sysName`/`name`.
  - 입력: 사번/이메일 1필드 + 비밀번호. `autoComplete` 유지.
  - 상태: 로딩(`busy`), 에러 메시지(계정/비번/잠금 구분), 엔터 제출.
  - 회사정보 조회: `companyInfoRepo.get()` 직접 호출(QueryProvider 밖이므로) 또는 Provider 재배치 후 훅.
  - 디자인 토큰: 기존 Tailwind `@theme` 토큰(navy/teal/panel/border 등) 재사용, 인라인 스타일 금지.

### 3.5 로그아웃 / 사용자 표시
- `UserMenu.tsx` — `currentUser.name`/`dept` 등 세션 정보 표기 반영(선택). 로그아웃 호출은 무변경.

## 4. 구현 순서

1. **도메인**: `user/schema.ts` password 추가 + seed 초기 비번 부여.
2. **데이터**: `data/auth/auth.repo.ts` 작성(authenticate/touchLastLogin).
3. **컨텍스트**: `AuthProvider.tsx` 자체 세션으로 재작성(localStorage 지속, API 유지).
4. **화면**: `LoginScreen.tsx` 회사로고 반영 신규 + `companyInfo` 연동.
5. **게이트**: `AuthGate.tsx` 확인/소폭 수정. Provider 순서 이슈 해결(repo 직접 호출 방식 채택).
6. **검증**: `npm run build 2>&1 | grep "error TS"` 0에러 확인 → 로컬 로그인/로그아웃/세션복원/실패케이스 수동 테스트.
7. **활성화**: `.env.local` `VITE_AUTH_ENABLED="true"` 로 전환(사용자 판단).
8. **commit/push** (사용자 요청 시).

## 5. 검토·확정 필요 사항

- [ ] 비밀번호 평문 저장 허용 여부(데모 한정) vs 최소 해시 적용? → 데모면 평문, 명시적 한계 주석.
- [ ] 로그인 식별자를 사번+이메일 둘 다로 할지, 하나로 고정할지.
- [ ] `lastLogin` 갱신 시 users 문서 write 발생 — 데모에서 켤지 여부.
- [ ] 세션 만료 정책(무기한 vs 시간제한). 기본: 브라우저 유지, 만료 없음.
- [ ] `demoMode`(Firebase 미설정) 동작 정책 — 자동 통과 유지 vs seed 계정 로그인 강제.

## 6. 영향 범위 / 리스크

- 기존 Firebase Auth 게이트 3파일 교체 → `firebase/auth` 의존 제거(다른 곳 미사용 확인 필요).
- `VITE_AUTH_ENABLED="true"` 전환 시점에만 로그인 화면 노출 → 기존 앱 흐름 무영향(현재 `false`).
- 보안룰은 개방형 유지 → 이 작업으로 permission-denied 리스크 없음.
- 진짜 인증은 후속(Cloud Function custom token) — 본 계획은 UI 게이트까지만.
