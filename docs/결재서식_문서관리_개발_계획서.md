## 0. 이 문서의 범위

전자결재 **문서 양식(서식/템플릿)을 코드 수정 없이 추가·수정·삭제·디자인**할 수 있는
관리 프로그램의 개발 계획. "결재선 규칙관리"가 *결재 흐름*을 데이터화했듯, 이 계획서는
*결재 문서 자체(유형·필드·인쇄양식)*를 데이터화한다.

- **대상**: 결재서식(문서유형) CRUD + 폼(필드) 디자이너 + 인쇄 레이아웃 + 상신/문서뷰 연동
- **비대상(이번 계획 아님)**: 결재 흐름(결재선)·전결규정(별도 완료), 리치텍스트 에디터, 외부 서식 파일(hwp/pdf) 임포트

---

## 1. 배경 · 현황

현재 전자결재는 **문서유형 4종이 코드에 하드코딩**되어 있고, 유형별 필드·인쇄양식이
세 곳에 흩어져 박혀 있다. 새 유형(출장신청서·구매요청서·교육신청서 등)을 추가하려면
개발자가 코드를 고쳐야 한다.

| 위치 | 하드코딩 내용 |
|---|---|
| `src/domain/approvalDoc/schema.ts` | `DOC_TYPES = ['기안','품의','지출결의','휴가']` 고정 enum · `form`은 `leaveFormSchema`(휴가 전용)만 |
| `src/modules/gw/approval/ApprovalDraftModal.tsx` | 유형 버튼 4개 · 금액필드(품의·지출결의) · 휴가필드(종류·기간) · 본문 textarea 를 `if`로 분기 |
| `src/modules/gw/approval/ApprovalDocumentView.tsx` | `DOC_TITLE`/`CLOSING` 유형별 맵 · 메타표·휴가상세표를 코드로 배치 |
| `src/data/seeds/approvalRoute.seed.ts` | 룰의 `docType`이 `DOC_TYPES` enum 을 참조(문서유형 확장 시 함께 열어야 함) |

즉, "문서유형·필드·격식(문서명/맺음말)·인쇄배치"가 전부 코드다. 이 계획은 이것을
**`approvalForms` 마스터**로 옮겨 **기준정보 화면에서 관리**하게 한다.

## 2. 목표 · 비목표

### 목표
1. **서식 CRUD**: 결재 문서유형(서식)을 화면에서 추가·수정·삭제·복제·활성/비활성.
2. **폼 디자이너**: 서식별 입력 필드를 정의(라벨·타입·필수·순서·2열 배치·섹션).
3. **격식 정의**: 격식 문서명(예: "출 장 신 청 서")·맺음말·아이콘·금액 사용여부.
4. **실시간 미리보기**: 편집 중 상신 폼과 인쇄 문서를 즉시 미리보기.
5. **무코드 연동**: 상신 모달·결재문서 뷰가 서식 정의를 읽어 **동적 렌더**(코드 분기 제거).
6. **하위호환**: 기존 4종(기안/품의/지출결의/휴가)·기존 문서·결재선 규칙이 그대로 동작.

### 비목표
- 픽셀 단위 자유 배치(절대좌표) · 리치텍스트/이미지 본문 에디터.
- 서식별 완전 독립 워크플로(결재선은 기존 룰엔진 재사용).
- 외부 문서(hwp/word) 변환·전자서명(PKI)·법적 원본성 보증.

## 3. 핵심 개념 — "결재서식(ApprovalForm)"

> **결재서식 = 문서유형(격식) + 입력 필드 정의 + 레이아웃**

하나의 서식은 (1) 상신 시 사용자가 채우는 **입력 폼**과 (2) 결재/인쇄 시 보이는 **격식 문서**
두 얼굴을 하나의 정의에서 생성한다. 기존 "휴가=leaveForm" 특례를 일반화한 것이다.

- 사용자는 서식 목록에서 유형을 고르고 → 서식이 정의한 필드를 채우고 → 결재선을 지정해 상신.
- 결재/인쇄 뷰는 같은 서식 정의로 메타표·상세표·본문 영역을 그린다.
- 결재선 규칙(route rule)은 서식의 `code`(문서유형)로 매칭 → 흐름은 기존 엔진 그대로.

## 4. 데이터 모델

### 4.1 `approvalForms` — 결재서식 마스터 (신규 컬렉션)

```ts
// src/domain/approvalForm/schema.ts (신규)
export const FIELD_TYPES = [
  '텍스트','장문','숫자','금액','날짜','기간','선택','다중선택','체크','사용자','부서','안내문',
] as const; // '기간' = 시작~종료(휴가·출장 등), '금액' = route rule 금액매칭 키

export const formFieldSchema = z.object({
  key: z.string().min(1),          // 문서 내 필드 식별자(영문/스네이크). 값 저장 키.
  label: z.string().min(1),        // 표시 라벨(예: '출장지')
  type: z.enum(FIELD_TYPES),
  required: z.boolean().default(false),
  options: z.array(z.string()).default([]),   // 선택/다중선택 항목
  placeholder: z.string().default(''),
  width: z.enum(['half','full']).default('full'), // 2열 배치
  section: z.string().default(''),  // 섹션(구분 제목) — 같은 값끼리 묶음
  isAmountKey: z.boolean().default(false), // 이 필드가 결재선 금액매칭 amount 로 쓰임(서식당 0~1개)
  order: z.number().default(0),
});

export const approvalFormSchema = z.object({
  id: z.string().min(1),           // PK = code
  code: z.string().min(1),         // 문서유형 코드(결재선 룰 docType 과 매칭). 예: '기안','출장'
  name: z.string().min(1),         // 표시명(예: '출장신청서')
  icon: z.string().default('📄'),
  docTitle: z.string().default(''),// 격식 문서명(예: '출 장 신 청 서'). 비면 name 사용
  closing: z.string().default(''), // 맺음말(예: '위와 같이 신청하오니 …')
  fields: z.array(formFieldSchema).default([]),
  active: z.boolean().default(true),
  order: z.number().default(0),
  system: z.boolean().default(false), // 기본 4종 보호 플래그(삭제 방지, 복제는 허용)
});
```

### 4.2 `approvalDocs` 일반화 (기존 스키마 확장, 하위호환)

- **추가**: `fieldValues: z.record(z.string(), z.union([z.string(), z.number(), z.boolean()])).default({})`
  — 서식 필드값을 `{ key: value }` 로 저장(휴가 `form` 특례를 대체·흡수).
- **`docType`**: 기존 enum → **문자열(서식 code)** 로 완화. 기본 4종 값은 그대로라 기존 문서·룰과 호환.
- **`form`(leaveForm)**: 당분간 유지(읽기 하위호환) → 휴가 서식 필드로 이관 후 deprecate.
- **`amount`**: 유지. 서식의 `isAmountKey` 필드값을 상신 시 `amount` 로 승격(결재선 금액매칭 유지).

> ⚠ `fieldValues` 는 **배열이 아닌 맵**이라 Firestore 중첩배열 제약 무관([[이슈_Firestore_중첩배열_코덱.md]]).
> RDB 이관 시 `approvalForms`=서식헤더, `formFields`=라인, `approvalDocFieldValues`=값 라인으로 분해([[DB_이관_대비_설계원칙.md]]).

### 4.3 결재선 룰 `docType` 완화

`approvalRouteRuleSchema.docType` 을 `enum(DOC_TYPES) | '전체'` → **`string`('전체' 포함)** 로 완화.
신규 서식 코드로도 룰을 매칭할 수 있게 한다(엔진 로직은 문자열 비교라 무변경).

## 5. 계층 설계 (정본 패턴 준수, [[데이터_계층_구현현황.md]])

```
domain/approvalForm/schema.ts          # SSOT (위 4.1)
```

- **연동 지점(수정)**: `ApprovalDraftModal`(동적 폼 렌더), `ApprovalDocumentView`(동적 인쇄),
  `approvalDoc.repo`(fieldValues·amount 승격), `approvalRoute/schema`(docType 완화).
- **Firestore 실 백엔드**: 신규 컬렉션 `approvalForms` 는 seed 후 `npm run seed --only=approvalForms` 로 적재.
  firestore.rules 에 컬렉션 추가 필요. (참고: 앱은 Firestore 를 실 백엔드로 읽음)

## 6. UI 설계 (기존 마스터-디테일 톤, 결재선 규칙관리와 동형)

### 6.1 서식 관리 화면 `/base/approval-form` (기준정보 › 조직·결재 기준정보)

3분할 — 좌: 서식 목록 / 중: 편집기 / 하단: 미리보기.

- **목록**: 아이콘·이름·code·필드수·활성. [+ 서식 추가] · 복제 · 삭제(system 서식은 삭제 잠금).
- **편집기**:
  - *기본*: 이름·code·아이콘·격식 문서명·맺음말·활성.
  - *필드 빌더*: 필드 행(라벨·타입·필수·2열/전체·섹션·옵션)을 추가/삭제/▲▼ 재정렬.
    타입에 따라 옵션 입력칸 노출(선택/다중선택), '금액' 필드는 결재선 금액키 지정 토글.
  - *레이아웃*: 섹션·2열 배치를 미리보기에 즉시 반영.
- **미리보기(2탭)**: ① 상신 폼 미리보기(실제 입력 UI), ② 인쇄 문서 미리보기(격식 A4 뷰).

### 6.2 상신 모달 동적화 (`ApprovalDraftModal` 리팩터)

- 유형 버튼을 **활성 서식 목록**에서 생성(하드코딩 4개 제거).
- 선택 서식의 `fields` 를 **동적 렌더**(타입별 위젯) → 값은 `fieldValues` 로 수집.
- `isAmountKey` 필드값을 `amount` 로 승격(결재선 룰·전결규정 그대로 작동).
- 결재선 빌더·자동 상신선(룰엔진)은 무변경(서식 code = docType 전달).

### 6.3 결재문서 뷰 동적화 (`ApprovalDocumentView` 리팩터)

- `DOC_TITLE`/`CLOSING` 하드맵 → 서식의 `docTitle`/`closing` 사용.
- 메타표 뒤 **상세표를 fieldValues×필드정의로 생성**(라벨·값, half/full 배치).
- 우상단 결재란 도장 grid·본문 영역·서명부는 기존 유지(서식 무관 공통 격식).

## 8. 기존 4종 마이그레이션 전략 (무중단·하위호환)

1. **시드 이관**: 기안·품의·지출결의·휴가를 `approvalForm.seed.ts` 에 `system:true` 서식으로 정의
   (지출결의/품의=금액 필드, 휴가=기간+종류 필드).
2. **읽기 호환**: 기존 문서의 `form`(leaveForm)·고정 필드는 뷰에서 계속 읽되, 신규 문서는 `fieldValues` 사용.
3. **docType 유지**: 4종 code 값을 그대로 두어 결재선 규칙 30개·전결규정·기존 문서 매칭 불변.
4. **점진 제거**: `DOC_TYPES` 상수는 "기본 코드"로 축소 참조, 이후 서식 마스터로 완전 대체.

## 9. 단계별 로드맵

| Phase | 범위 | 산출 | 난이도 |
|---|---|---|---|
| **P1 · 서식 엔진 + 폼 빌더(MVP)** | approvalForm 스키마·repo·훅·관리화면(필드 CRUD·미리보기), 상신 모달 동적화, 4종 시드 이관 | 화면에서 새 서식 추가→상신 가능 | ★★★ |
| **P2 · 문서 디자인(인쇄)** | 문서뷰 동적화(격식·상세표), 섹션·2열 레이아웃, 인쇄 미리보기 | 서식별 인쇄 문서 완성 | ★★ |
| **P3 · 고급** | 드래그 재정렬, 표(라인아이템) 필드, 조건부/수식 필드, 서식 버전·복제, 결재선 규칙 자동연결 | 복잡 서식 대응 | ★★★★ |

권장: **P1 → P2 를 1차 목표**(실사용 가능), P3 는 필요 시 확장.

## 10. 리스크 · 대응

| 리스크 | 대응 |
|---|---|
| `docType` enum→문자열 완화가 룰엔진·시드·타입에 파급 | 4종 코드 불변 유지 + `string` 완화만(비교 로직 무변경), 타입체크로 회귀 확인 |

| Firestore 실 백엔드 반영 누락 | 신규 컬렉션 seed + rules 등록, `npm run seed --only=approvalForms` 절차 문서화 |

| 폼 빌더 과설계(P3까지 한 번에) | P1은 "필드 목록 + 2열 배치"까지로 스코프 고정 |

## 11. 결정 필요 사항 (구현 착수 전 확인)

1. **디자인 깊이**: (A) 필드 목록·2열 배치·미리보기(P1+P2 권장) / (B) 드래그앤드롭·표·수식까지(P3).
2. **신규 유형 허용 범위**: (A) 기존 4종 + **자유 신규 서식 추가**(범용 엔진, 권장) / (B) 기존 4종 필드 커스터마이즈만(경량).
3. **인쇄 레이아웃**: (A) 라벨-값 자동 표 생성(권장) / (B) 서식별 배치 세밀 조정.
4. **결재선 자동연결**: 신규 서식 추가 시 기본 결재선 규칙도 함께 만들지(P3) — 우선은 수동 지정.

> 위 4개(특히 1·2)에 따라 P1 범위와 일정이 결정된다. 기본 권장안은 **2-A + 1-A**
> (자유 서식 추가 + 필드/2열/미리보기)로, 실사용 가능한 최소 완결 형태.
